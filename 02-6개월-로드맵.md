## 주간 리듬 (권장)

완주율을 가르는 건 총 시간이 아니라 리듬이다.

| 요일 | 시간 | 내용 |
|---|---|---|
| 월·화·수 | 각 1시간 | **읽기/개념** — 문서, 강의, 명세 |
| 목 | 1시간 | **손으로 짜기** — 그 주 마이크로 실습 |
| 금 | 1시간 | **정리** — 그 주 요약 1장 (자기 말로) |
| 주말 | 0~2시간 | 밀린 것 보충 / 여유 |

**금요일의 요약 1장이 이 로드맵에서 가장 중요한 습관이다.**
완료 기준이 "설명할 수 있다"인데, 그 판정을 매주 자동으로 해주는 장치가 이 글쓰기다.

> 한 주 빠지면 로드맵이 1주 밀릴 뿐 실패가 아니다.
> 6개월짜리 계획에서 진짜 위험은 지연이 아니라 **중단**이다.

## Stage 1 — HTTP와 웹의 동작 원리 (4주)

> 문서: [03-Stage 1 — HTTP와 웹의 동작 원리](03-stage1-HTTP와-웹의-동작-원리.md)

**핵심 질문**: 주소창에 URL을 치면 정확히 무슨 일이 일어나는가?

백엔드 교양의 뿌리이자 **가장 오래 남는 지식**이다. 서두르지 않는다.

| 주차 | 내용 | 마이크로 실습 |
|---|---|---|
| 1주 | 네트워크 계층, IP/TCP, DNS, 포트 | `curl -v`로 요청/응답 원문 뜯어보기 |
| 2주 | HTTP 메시지 구조, 메서드, 상태코드, 헤더 | `node:net`으로 TCP 소켓에 HTTP 손으로 쓰기 |
| 3주 | 쿠키, 캐시, CORS, HTTPS/TLS | 브라우저 개발자도구 Network 탭 완독 |
| 4주 | HTTP/1.1 vs 2 vs 3, Keep-Alive, 첫 서버 | `node:http`로 서버 만들기 (30줄) |

**완료 기준 (말로 설명 가능해야 함)**
- 401과 403의 차이, 그리고 왜 그 구분이 존재하는가
- CORS 에러가 브라우저에서만 나고 curl에서는 안 나는 이유
- 쿠키의 `HttpOnly`, `Secure`, `SameSite`가 각각 무엇을 막는가
- GET과 POST의 진짜 차이 (파라미터 위치가 아니다)

## Stage 2 — Node.js 런타임과 비동기 (4주)

> 문서: [04-Stage 2 — Node.js 런타임과 비동기](04-stage2-Node-런타임과-비동기.md)

**핵심 질문**: 스레드 하나로 어떻게 수천 명을 동시에 받는가?

Node의 강점과 약점을 동시에 만드는 **단 하나의 설계 결정**을 이해하는 스테이지다.

| 주차 | 내용 | 마이크로 실습 |
|---|---|---|
| 1주 | 이벤트 루프, 논블로킹 I/O, 단일 스레드의 의미 | 블로킹 코드로 서버 멈추는 것 재현 |
| 2주 | Promise, async/await, 에러 전파, `Promise.all` | 순차 vs 병렬 요청 시간 비교 |
| 3주 | 모듈 시스템, npm, 패키지 고르는 법 | `package.json` 손으로 읽기 |
| 4주 | 스트림, 백프레셔, `node:` 내장 모듈 | 큰 파일 스트리밍 응답 |

**완료 기준**
- 이벤트 루프가 무엇이고 왜 그렇게 설계됐는가
- CPU 무거운 작업을 Node 서버에서 하면 왜 위험한가
- `await` 3개를 순차로 vs `Promise.all`로 했을 때 시간 차이와 그 이유
- npm 패키지를 추가하기 전에 무엇을 확인해야 하는가

### ▶ P1 실습 — 프레임워크 없는 서버 (+4시간)

## Stage 3 — 데이터베이스와 SQL (4주)

> 문서: [05-Stage 3 — 데이터베이스와 SQL](05-stage3-데이터베이스와-SQL.md)

**핵심 질문**: 상태를 왜 밖에 두는가? 동시에 쓰면 어떻게 되는가?

[01-백엔드의 5가지 근본 전제](01-백엔드의-5가지-근본-전제.md)의 제1계율(stateless)이 필연적으로 도착하는 곳이다.

**⚠️ 3주차(트랜잭션·동시성)가 이 스테이지의 심장이다.** SQL 문법보다 여기에 시간을 쓴다.

| 주차 | 내용 | 마이크로 실습 |
|---|---|---|
| 1주 | 관계형 모델, 정규화, 키, 스키마 설계 | 스키마를 종이에 그려보기 |
| 2주 | SQL: SELECT/JOIN/GROUP BY, 인덱스 | 인덱스 유무 실행계획 비교 |
| 3주 | **트랜잭션, ACID, 격리 수준, 락** | 동시 UPDATE로 갱신 유실 재현 |
| 4주 | ORM/쿼리빌더, N+1, 마이그레이션 | N+1 재현 후 배치 쿼리로 수정 |

**완료 기준**
- 트랜잭션이 필요한 상황을 계좌이체 말고 다른 예로 설명할 수 있다
- 격리 수준을 낮추면 무엇을 얻고 무엇을 잃는가
- 인덱스를 무조건 많이 걸면 안 되는 이유
- N+1 문제가 무엇이고 어떻게 없애는가

## Stage 4 — API 설계 (4주)

> 문서: [06-Stage 4 — API 설계](06-stage4-API-설계.md)

**핵심 질문**: 좋은 API와 나쁜 API를 가르는 것은 무엇인가?

여기는 **코드 지식이 아니라 판단력**이다. 정답이 없어서 오히려 어렵다.

| 주차 | 내용 | 마이크로 실습 |
|---|---|---|
| 1주 | 리소스 모델링, REST의 진짜 의미, URL 설계 | 나쁜 API 5개 찾아서 왜 나쁜지 쓰기 |
| 2주 | 상태코드 규약, 에러 응답 설계, 입력 검증 | 에러 응답 포맷 정해서 문서화 |
| 3주 | 페이지네이션, 필터링, 버저닝, 멱등성 | 커서 vs 오프셋 페이지네이션 비교 |
| 4주 | **프레임워크 도입**, 미들웨어, 스키마 검증 | P1 서버를 프레임워크로 재작성 |

**완료 기준**
- REST에서 "리소스"가 왜 동사가 아니라 명사인가
- 클라이언트가 화면 하나 그리는 데 API 5번 호출해야 한다면 무엇이 잘못됐나
- API 버저닝이 필요해지는 순간은 언제인가
- 미들웨어 파이프라인에서 순서가 왜 중요한가

### ▶ P2 실습 — 인증 있는 API 서버 (+10시간)

## Stage 5 — 인증과 보안 (4주)

> 문서: [07-Stage 5 — 인증과 보안](07-stage5-인증과-보안.md)

**핵심 질문**: 서버는 상대가 누구인지 어떻게 아는가?

**가장 헷갈리는 스테이지**다. 용어가 겹치고(인증/인가/세션/토큰/JWT/OAuth),
인터넷 자료의 상당수가 오래됐거나 틀렸다. 개념 정리에 시간을 넉넉히 쓴다.

| 주차 | 내용 | 마이크로 실습 |
|---|---|---|
| 1주 | 인증 vs 인가, 비밀번호 저장(해싱/솔트) | bcrypt로 해싱, 왜 SHA-256이 아닌지 확인 |
| 2주 | 세션 기반 인증, 쿠키 보안 속성 | 쿠키 세션 로그인 30줄로 구현 |
| 3주 | JWT의 구조·장단점, 리프레시 토큰 | JWT 직접 디코딩해서 내용 보기 |
| 4주 | OAuth2/OIDC 개념, 웹 취약점 (XSS·CSRF·SQLi) | SQL 인젝션 재현 후 바인딩으로 방어 |

**완료 기준**
- 인증과 인가의 차이를 예시로 설명할 수 있다
- 왜 비밀번호는 암호화가 아니라 해싱인가
- JWT가 세션보다 무조건 좋은 게 아닌 이유
- XSS와 CSRF가 각각 무엇을 노리고, 무엇으로 막는가

## 이 로드맵을 버려도 되는 때

계획은 도구지 계약이 아니다. 아래에 해당하면 **로드맵을 접고 그쪽으로 간다.**

1. **실무에서 백엔드 과제가 실제로 떨어졌을 때** — 실전이 항상 로드맵을 이긴다
2. **만들고 싶은 것에 서버가 당장 필요해졌을 때**
   → 필요한 스테이지만 먼저 뽑아 쓴다. 순서를 지키는 것보다 만드는 게 낫다
3. **특정 주제에 불이 붙었을 때** — 재미가 있는 곳에서 학습 효율이 몇 배다

단, ①~③ 중 어느 경우든 **Stage 1(HTTP)만은 건너뛰지 않는다.**
나머지는 나중에 메꿀 수 있지만, HTTP를 모른 채 쌓은 것은 전부 모래 위에 짓는 것이다.
