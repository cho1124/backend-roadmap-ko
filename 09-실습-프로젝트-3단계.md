# 09 - 실습 프로젝트 3단계

> 허브: [00-백엔드 학습 허브](00-백엔드-학습-허브.md)
> 원칙: **각 프로젝트는 앞선 스테이지들을 하나로 묶는 지점.** 새 개념을 배우는 곳이 아니다.

---

## 왜 프로젝트를 3개만 하는가

원리 이해가 목표이므로 **큰 제품을 만드는 것이 목적이 아니다.**
스테이지마다 배운 마이크로 실습들이 **서로 연결되는 경험**을 3번만 하면 충분하다.

```
Stage 1~2 끝  →  P1  프레임워크 없는 서버   (~4시간)
Stage 3~4 끝  →  P2  인증 있는 API 서버     (~10시간)
Stage 6   끝  →  P3  배포와 운영            (~6시간)
```

**P1과 P3는 버리는 코드**다. P2만 남긴다.

---

## P1 — 프레임워크 없는 서버 (Stage 2 직후, ~4시간)

### 목적

> **프레임워크가 무엇을 대신해주는지 알기 위해, 먼저 없이 만들어본다.**

Stage 4에서 프레임워크로 재작성할 때, 사라진 코드가 곧 프레임워크의 가치다.

### 만들 것: URL 단축기 (메모리 저장)

```
POST /links        { "url": "https://..." }  → { "id": "a3f", "shortUrl": "..." }
GET  /links/:id                              → 원본 URL 정보
GET  /:id                                    → 301 리다이렉트
GET  /health                                 → { "status": "ok" }
```

**DB 없이 `Map`으로 저장한다.** Stage 3을 아직 안 했으므로.
(그리고 재시작하면 날아가는 걸 겪는 것이 Stage 3의 동기가 된다.)

### 제약 (이게 핵심이다)

```
✅ 허용: node: 내장 모듈만
❌ 금지: express, fastify, hono 등 웹 프레임워크
❌ 금지: 라우팅/파싱 라이브러리
```

### 직접 만들어야 하는 것들

- [ ] 라우팅 — `req.method` + `req.url` 매칭 (경로 파라미터 `:id` 포함)
- [ ] 요청 본문 읽기 — 스트림에서 청크를 모아 JSON 파싱
- [ ] 쿼리스트링 파싱 — `URL` 클래스 활용
- [ ] JSON 응답 헬퍼 — 상태코드 + `Content-Type` + `charset`
- [ ] 에러 처리 — 잘못된 JSON, 없는 경로, 잘못된 URL
- [ ] 요청 로깅 — 메서드·경로·상태·소요시간

### 완료 기준

- [ ] `curl`로 전 기능 동작
- [ ] 잘못된 JSON을 보내면 서버가 **죽지 않고** 400 반환
- [ ] 없는 경로는 404
- [ ] 리다이렉트가 브라우저에서 실제로 동작
- [ ] 요청 로그가 남는다

### 회고 (반드시 적는다)

> **"무엇이 가장 귀찮았는가?"**

이 목록을 남긴다. Stage 4에서 프레임워크를 도입할 때
**그 목록이 하나씩 지워지는 걸 확인**하는 것이 이 프로젝트의 진짜 결과물이다.

---

## P2 — 인증 있는 API 서버 (Stage 4 직후, ~10시간)

### 목적

> **Stage 3(DB)과 Stage 4(API 설계)를 하나로 묶는다.** Stage 5 이후 인증을 보강한다.

### 주제 고르기 (30분 이상 쓰지 않는다)

무엇을 만들든 상관없다. **아래 조건만 만족하면 된다.**

```
✅ 사용자 계정이 있다        (인증·인가 연습)
✅ 사용자별 데이터가 있다     (소유권 검사 연습)
✅ 목록 조회가 있다          (페이지네이션 연습)
✅ 항목에 태그/카테고리가 붙는다 (N:M 관계·N+1 연습)
✅ 테이블 3~4개 규모        (너무 커지지 않게)
```

**무난한 주제 예시:**

| 주제 | 특징 |
|---|---|
| **북마크 저장소** | 가장 단순. URL + 제목 + 태그. CRUD 연습에 최적 |
| **메모 / 일기 앱 API** | 사용자별 기록 + 태그. 검색·기간 필터가 자연스럽다 |
| **할 일 관리** | 상태 전이(완료/미완료)가 있어 PATCH 설계 연습에 좋다 |
| **간단한 게시판** | 댓글까지 넣으면 1:N 중첩 관계를 다뤄볼 수 있다 |

> **이미 만들고 싶은 게 있으면 그걸로 한다.** 동기가 있는 쪽이 항상 낫다.
> 다만 위 5개 조건을 만족하도록 스코프를 조정한다.

### 스코프 (여기서 벗어나지 않는다)

아래는 "북마크 저장소" 기준. 다른 주제면 명사만 바꾼다.

```
[ 인증 ]
POST   /auth/register          가입
POST   /auth/login             로그인 (세션 or 토큰)
POST   /auth/logout            로그아웃

[ 리소스 ]
GET    /bookmarks              목록 (커서 페이지네이션 + 태그 필터)
POST   /bookmarks              생성
GET    /bookmarks/:id          단건
PATCH  /bookmarks/:id          수정
DELETE /bookmarks/:id          삭제

[ 부가 ]
GET    /tags                   태그 목록
GET    /health
```

### 명시적 비스코프 (하지 않는다)

```
❌ 프론트엔드 화면
❌ 소셜 로그인
❌ 파일 업로드
❌ 실시간 기능
❌ 검색 엔진 연동
❌ 관리자 기능
```

**스코프 확장이 학습 프로젝트를 죽이는 1순위 원인이다.**
재밌어져서 기능을 늘리기 시작하면 로드맵이 무너진다.

### 체크리스트

**Stage 3 적용:**
- [ ] 스키마 설계 (users, bookmarks, tags, bookmark_tags)
- [ ] 마이그레이션 파일로 관리
- [ ] 조회 조건 컬럼에 인덱스
- [ ] 여러 테이블을 함께 바꾸는 작업은 트랜잭션으로
- [ ] N+1 없이 태그를 함께 조회

**Stage 4 적용:**
- [ ] 리소스 중심 URL, 적절한 메서드
- [ ] 상태코드 정확히 (201, 204, 404, 409, 422)
- [ ] 통일된 에러 포맷 + `code` + `traceId`
- [ ] 스키마 기반 입력 검증
- [ ] 커서 페이지네이션
- [ ] 계층 분리 (route / service / repository)

**Stage 5 적용 (Stage 5 이후 보강):**
- [ ] 비밀번호 bcrypt/argon2
- [ ] 세션 또는 토큰 인증
- [ ] 쿠키 보안 속성
- [ ] **소유권 검사** — 남의 데이터에 접근 불가 (IDOR 방지)
- [ ] 파라미터 바인딩 (SQL 인젝션 방지)
- [ ] 로그인 레이트 리밋

### 완료 기준

- [ ] 두 계정을 만들어서 **서로의 데이터가 안 보이는 것 확인**
- [ ] 잘못된 입력에 대해 필드별 에러가 나옴
- [ ] 데이터 1,000건에서도 목록 조회가 빠름
- [ ] 서버를 재시작해도 데이터가 남아 있음
- [ ] **API 문서가 있다** (OpenAPI 또는 마크다운)

### 회고

> **"이 API를 클라이언트 개발자로서 쓴다면 무엇이 불편한가?"**

[06-Stage 4 — API 설계](06-stage4-API-설계.md) 1주차에서 만든 "나쁜 API 5개" 목록과 대조한다.
내가 만든 API가 그 목록에 들어가지 않는지 확인하는 것이 요점이다.

## P3 — 배포와 운영 (Stage 6, ~6시간)

### 목적

> **P2를 실제로 돌아가는 서비스로 만든다.** 새 기능은 추가하지 않는다.

### 체크리스트

**설정:**
- [ ] 모든 설정을 환경변수로
- [ ] 부팅 시 설정 검증 (없으면 즉시 종료)
- [ ] `.env.example` 작성, `.env`는 gitignore

**컨테이너:**
- [ ] 멀티스테이지 Dockerfile
- [ ] `USER node` (비root)
- [ ] `docker compose`로 앱 + DB
- [ ] 이미지 크기 확인

**관측:**
- [ ] 구조화 로깅 (JSON)
- [ ] traceId 미들웨어 + 응답 헤더
- [ ] 요청 소요시간 로깅
- [ ] 민감 필드 마스킹
- [ ] `/health`, `/ready`

**배포:**
- [ ] 실제 서버 또는 VM에 배포
- [ ] 리버스 프록시 + HTTPS
- [ ] 우아한 종료 (SIGTERM)
- [ ] CI 파이프라인 (테스트 → 빌드 → 배포)

**백업:**
- [ ] DB 자동 백업
- [ ] **복구를 실제로 해본다**

### 완료 기준 — 실제로 해보는 3가지

이 세 개가 P3의 전부다. 문서만 읽고 끝내면 아무것도 안 배운다.

```
① 배포 중 요청 끊김 테스트
   배포하면서 계속 요청을 보낸다 → 우아한 종료 전/후 비교

② 장애 재현과 추적
   일부러 느린 쿼리를 넣는다 → 로그·지표로 찾아낸다 → 고친다

③ 데이터 복구
   DB를 일부러 날린다 → 백업으로 복구한다
```

③은 심리적으로 하기 싫지만 **가장 가치 있는 실습**이다.
복구해본 적 없는 백업은 백업이 아니다.

### 회고

> **"이 서비스가 새벽 3시에 죽으면, 나는 무엇을 보고 원인을 찾는가?"**

답이 안 나오는 부분이 곧 관측 가능성의 구멍이다.

## 프로젝트 진행 규칙

| 규칙 | 이유 |
|---|---|
| **스코프를 늘리지 않는다** | 학습 프로젝트를 죽이는 1순위 원인 |
| **완벽하게 만들지 않는다** | P1·P3는 버리는 코드다 |
| **막히면 30분 룰** | 30분 막히면 검색/AI. 단, **답이 아니라 왜를 묻는다** |
| **매 프로젝트 회고 1장** | 이게 실제 산출물이다. 코드가 아니라 |
| **git으로 관리** | 커밋 메시지로 사고 과정이 남는다 |

### AI 사용 원칙 (중요)

```
❌ "북마크 API 만들어줘"              → 코드는 나오지만 학습은 0
✅ "이 스키마 설계에서 놓친 게 뭘까?"   → 검토자로
✅ "왜 여기서 트랜잭션이 필요한가?"     → 설명자로
✅ "이 코드의 보안 문제를 찾아줘"       → 리뷰어로
```

**생성된 코드는 한 줄씩 설명할 수 있어야 통과다.**
설명 못 하는 줄이 있으면 그건 내 코드가 아니다.

---

## 결론

> 프로젝트의 산출물은 **코드가 아니라 회고**다.
>
> P1은 "프레임워크가 뭘 해주는지"를,
> P2는 "설계 판단이 왜 어려운지"를,
> P3는 "만드는 것과 운영하는 것이 다르다"는 것을 남긴다.
>
> 이 세 문장이 남으면 코드는 지워도 된다.
