## 1주차 — 설정 분리와 12-Factor

### 같은 코드가 여러 환경에서 돌아야 한다

```
개발(내 PC)   → SQLite 파일, 로그 상세, 포트 3000
스테이징      → 테스트 DB, 로그 보통
운영          → 운영 DB, 로그 최소, HTTPS, 포트 443
```

**코드는 하나여야 한다.** 환경마다 다른 건 **설정**뿐이다.

```ts
// ❌ 하드코딩
const db = connect('postgres://localhost:5432/dev');

// ❌ 코드 안에 분기 — 환경이 늘어나면 지옥
if (process.env.NODE_ENV === 'production') { ... } else { ... }

// ✅ 환경변수로 주입 + 시작 시점에 검증
const config = ConfigSchema.parse({
  databaseUrl: process.env.DATABASE_URL,
  port:        Number(process.env.PORT ?? 3000),
  logLevel:    process.env.LOG_LEVEL ?? 'info',
});
```

> **시작 시점 검증이 중요하다.** 설정이 빠졌으면 **즉시 죽어야 한다.**
> 30분 뒤 첫 DB 요청에서 죽는 것보다, 부팅 3초 만에 명확한 메시지로 죽는 게 낫다.
> **빨리 실패하기(fail fast)**가 운영에서 가장 값싼 안전장치다.

### 12-Factor App — 실제로 유용한 것만

12개 중 백엔드 교양으로 꼭 알아야 할 5개:

| 원칙 | 뜻 | 왜 |
|---|---|---|
| **설정을 환경에** | 코드에 설정 없음 | 같은 이미지를 모든 환경에 |
| **백엔드 서비스는 부착 자원** | DB·캐시를 URL로 교체 가능하게 | 이전이 쉬워짐 |
| **프로세스는 stateless** | 로컬 파일/메모리에 상태 저장 안 함 | 수평 확장·재시작 가능 |
| **로그는 이벤트 스트림** | 파일 관리하지 말고 stdout으로 | 수집은 인프라의 일 |
| **폐기 가능** | 빠른 시작, 우아한 종료 | 배포·확장이 빨라짐 |

**"로그를 파일로 쓰지 않는다"**가 처음엔 이상하게 들린다.
이유: 로그 로테이션·수집·전송은 **앱의 관심사가 아니다.**
앱은 stdout에 뱉고, 컨테이너 런타임이나 수집기가 알아서 가져간다.

### 시크릿 관리

```
❌ 코드에 API 키   → git 히스토리에 영구 박제 (지워도 남는다)
❌ .env를 커밋     → 같은 문제
✅ .env는 .gitignore, .env.example만 커밋
✅ 운영은 배포 플랫폼의 시크릿 기능 / 시크릿 매니저
```

**이미 유출됐다면 파일 삭제로는 부족하다 — 키를 폐기하고 재발급한다.**

### 🔧 마이크로 실습 (40분)

- [ ] 프로젝트의 모든 하드코딩 값을 환경변수로 추출
- [ ] 설정 스키마 검증 추가 → 필수 값 빼고 실행해서 **부팅 즉시 죽는지** 확인
- [ ] `.env.example` 작성, `.env`를 `.gitignore`에 추가

## 2주차 — 컨테이너화

### 왜 컨테이너인가

```
"내 PC에선 되는데요" 의 원인:
  Node 버전 차이 / OS 라이브러리 차이 / 환경변수 차이 / 설치된 도구 차이

컨테이너: 실행에 필요한 모든 것을 이미지 하나로 고정
        → 개발·스테이징·운영이 비트 단위로 같은 것을 실행
```

### Node용 Dockerfile — 멀티스테이지

```dockerfile
# --- 빌드 단계 ---
FROM node:24-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                     # lock 파일 그대로 (install 아님)
COPY . .
RUN npm run build

# --- 실행 단계 ---
FROM node:24-alpine
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev          # devDependencies 제외
COPY --from=builder /app/dist ./dist
USER node                      # ★ root로 실행하지 않는다
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

**포인트 5개:**

| 항목 | 이유 |
|---|---|
| 멀티스테이지 | 빌드 도구가 최종 이미지에 안 들어감 → 작고 안전 |
| `COPY package*.json` 먼저 | 의존성이 안 바뀌면 레이어 캐시 재사용 → 빌드 빨라짐 |
| `npm ci` | lock 그대로 재현. `npm install`은 버전이 바뀔 수 있다 |
| `--omit=dev` | 운영에 불필요한 것 제외 |
| `USER node` | **root로 돌리면 컨테이너 탈출 시 피해가 커진다** |

(Node 버전은 [11-2026 생태계 현황 스냅샷](11-2026-생태계-현황-스냅샷.md) 확인 후 LTS로 맞춘다.)

### 헬스체크와 우아한 종료

```ts
// 헬스체크 — 로드밸런서/오케스트레이터가 상태를 확인
app.get('/health',  () => ({ status: 'ok' }));                     // 프로세스 살아있나 (liveness)
app.get('/ready',   async () => { await db.ping(); return {...} }); // 요청 받을 준비 됐나 (readiness)
```

**둘을 나누는 이유**: 프로세스는 살아있지만 DB 연결이 끊겼다면 —
**재시작(liveness 실패)이 아니라 트래픽 제외(readiness 실패)**가 맞다.
재시작해도 DB가 살아나지는 않으니까.

```ts
// 우아한 종료 — 배포 시 처리 중인 요청을 죽이지 않는다
process.on('SIGTERM', async () => {
  server.close();                 // ① 새 요청 안 받음
  await inFlightRequests();       // ② 처리 중인 것 마무리
  await db.end();                 // ③ 커넥션 정리
  process.exit(0);
});
```

**이게 없으면 배포할 때마다 일부 사용자의 요청이 끊긴다.**
평소엔 티가 안 나서 방치되기 쉬운데, 결제 같은 데서 사고가 난다.

### 🔧 마이크로 실습 (1.5시간)

- [ ] Dockerfile 작성 → 빌드 → 실행
- [ ] 이미지 크기 확인, 멀티스테이지 전/후 비교
- [ ] `docker compose`로 앱 + DB 함께 띄우기
- [ ] `/health`, `/ready` 추가
- [ ] SIGTERM 핸들러 넣고 `docker stop` 시 로그로 종료 순서 확인

## 3주차 — 관측 가능성

### 세 기둥

```
로그(Logs)      "무슨 일이 있었나"    — 사건, 텍스트, 디버깅용
메트릭(Metrics)  "얼마나/얼마나 빨리"  — 숫자 시계열, 대시보드·알림용
트레이스(Traces) "어디를 거쳤나"      — 한 요청의 전체 경로
```

셋은 답하는 질문이 다르다. 하나로 나머지를 대신할 수 없다.

### 구조화 로깅 — 문자열이 아니라 데이터로

```ts
// ❌ 사람만 읽을 수 있음. 검색·집계 불가
console.log(`유저 ${userId}가 ${postId} 조회, ${ms}ms 걸림`);

// ✅ 기계가 질의할 수 있음
logger.info({ event: 'post.viewed', userId, postId, durationMs: ms, traceId });
```

JSON 로그면 이런 게 가능해진다:
```
"durationMs > 1000 이고 event=post.viewed 인 것"
"traceId=req-8f3a2b 인 로그 전부"          ← 한 요청의 전 과정 추적
"최근 1시간 에러를 event별로 집계"
```

### 로그 레벨과 절대 금지 항목

| 레벨 | 언제 |
|---|---|
| `error` | 사람이 봐야 함. 알림 대상 |
| `warn` | 이상하지만 처리는 됨 |
| `info` | 주요 사건 (시작, 배포, 로그인) |
| `debug` | 개발 중에만 |

```
🚫 로그에 절대 남기면 안 되는 것
   비밀번호 · 토큰/세션ID · 카드번호 · 주민번호 ·
   요청 본문 통째로(개인정보 포함 가능) · Authorization 헤더
```

**로그는 보존되고 여러 사람이 본다.** 실질적으로 DB보다 유출 위험이 높은 경우도 있다.
→ 민감 필드 마스킹을 **로거 설정에 넣어서** 실수를 구조적으로 막는다.

### 요청 추적 (correlation)

```ts
// 요청마다 ID를 만들고 모든 로그에 붙인다
app.use((req, res, next) => {
  req.traceId = req.headers['x-request-id'] ?? crypto.randomUUID();
  res.setHeader('x-request-id', req.traceId);
  next();
});
```

응답 헤더에도 넣으면 — 사용자가 "에러 났어요" 하며 캡처를 보내왔을 때
**그 ID로 로그를 즉시 찾을 수 있다.** Stage 4의 에러 응답 `traceId`와 연결된다.

### 봐야 할 메트릭 (골든 시그널)

| 지표 | 뜻 |
|---|---|
| **Latency** | 응답 시간. **평균 말고 p95/p99** |
| **Traffic** | 초당 요청 수 |
| **Errors** | 에러율 (5xx 비율) |
| **Saturation** | 자원 포화도 (CPU·메모리·커넥션 풀) |

**왜 평균이 아니라 p99인가:**
```
평균 50ms, p99 3초인 서비스
→ 100명 중 1명은 3초를 기다린다
→ 페이지에 API 호출이 10개면, 사용자 10명 중 1명은 매번 3초짜리를 만난다
```

평균은 **꼬리(tail)를 숨긴다.** 그리고 사용자 체감은 꼬리가 지배한다.

### "느려졌어요" 신고를 받으면

```
① 언제부터? → 배포 시각과 대조 (대부분 여기서 끝난다)
② 전체인가 일부인가? → 특정 엔드포인트만? 특정 사용자만?
③ 지표 확인 → p99 상승? 에러율 상승? CPU/메모리?
④ 로그 확인 → 느린 요청의 traceId로 전 과정 추적
⑤ 흔한 원인 → 느린 쿼리 / N+1 / 커넥션 풀 고갈 / 외부 API 지연 / 메모리 누수
```

**①이 가장 자주 정답이다.** 배포와 장애의 상관관계를 먼저 본다.

### 🔧 마이크로 실습 (1.5시간)

- [ ] `console.log`를 구조화 로거(pino 등)로 교체
- [ ] traceId 미들웨어 추가 → 한 요청의 로그가 전부 묶이는지 확인
- [ ] 요청 소요 시간을 로그에 기록
- [ ] 민감 필드 마스킹 설정
- [ ] 일부러 느린 엔드포인트를 만들고 로그로 찾아내기

## 4주차 — 배포

### 앞단 구성

```
인터넷 → [리버스 프록시 (Nginx/Caddy)] → [Node 앱 × N]
              ↑
        TLS 종료 · 정적 파일 · 압축 · 로드밸런싱 · 레이트리밋
```

**Node 앱이 직접 443 포트를 잡지 않는 이유**: TLS 인증서 관리, 정적 파일 서빙,
압축 같은 건 프록시가 훨씬 잘한다. Node는 애플리케이션 로직에만 집중한다.

### 프로세스 관리와 수평 확장

```
문제: Node는 CPU 코어 하나만 쓴다 (Stage 2)
해결: 코어 수만큼 프로세스를 띄운다

  - PM2 / systemd    단일 서버
  - Docker replicas  컨테이너
  - 오케스트레이터    여러 서버
```

**전제 조건**: 앱이 stateless여야 한다.
프로세스가 메모리에 세션을 들고 있으면 요청이 다른 프로세스로 갈 때 로그아웃된다.
→ **Stage 5의 세션 저장소를 밖으로 빼는 이유가 여기서 완성된다.**

### 배포 전략

| 전략 | 방식 | 다운타임 |
|---|---|---|
| **재시작** | 끄고 켠다 | 있음 (소규모는 이걸로 충분) |
| **롤링** | 인스턴스를 하나씩 교체 | 없음 |
| **블루/그린** | 새 환경 준비 후 트래픽 전환 | 없음, 롤백 빠름 |
| **카나리** | 5% → 50% → 100% 점진 | 없음, 위험 최소 |

**무중단 배포가 어려운 진짜 이유**: 배포 순간 **구/신 버전이 동시에 돈다.**
→ DB 스키마 변경이 양쪽 모두와 호환되어야 한다.

```
❌ 컬럼 이름 변경 한 방에      → 구버전이 즉시 깨진다
✅ ① 새 컬럼 추가 (양쪽 호환)
   ② 새 버전 배포 (새 컬럼 사용)
   ③ 구 컬럼 제거 (다음 배포에서)
```

이 **"확장 → 이전 → 축소"** 패턴이 무중단 스키마 변경의 핵심이다.
Stage 3의 마이그레이션 원칙("적용된 것은 고치지 않고 새로 추가")과 같은 사고다.

### CI/CD

```yaml
push → 테스트 → 린트 → 빌드 → 이미지 푸시 → 배포
```

**게이트를 어디에 두느냐가 설계다:**
- 테스트 실패 → 배포 중단 (당연)
- main 브랜치만 운영 배포
- 운영 배포는 수동 승인 (소규모라도 권장)

### 백업 — 가장 자주 잊히는 것

```
백업이 없는 서비스는 "언젠가 데이터를 잃는" 서비스다.
그리고 복구를 테스트하지 않은 백업은 백업이 아니다.
```

- [ ] DB 자동 백업 (일 1회 이상)
- [ ] 백업을 다른 곳에 보관
- [ ] **복구를 실제로 한 번 해본다** ← 이걸 안 하는 경우가 대부분

### 🔧 마이크로 실습 (2시간)

- [ ] 프로젝트를 실제 서버 또는 로컬 VM에 배포
- [ ] 리버스 프록시 앞에 두고 HTTPS 적용
- [ ] CI 파이프라인 (테스트 → 빌드 → 배포)
- [ ] 배포 중 요청을 계속 보내며 끊김 관찰 → 우아한 종료 효과 확인
- [ ] DB 백업 → **일부러 지우고 복구**
