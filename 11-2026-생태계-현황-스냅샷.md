## Node.js 버전

| 항목 | 2026-07 현재 |
|---|---|
| **Current** | Node 26 (2026-05-05 릴리스, 2026-10 LTS 진입 예정) |
| **Active LTS** | **Node 24** (2028-04-30까지 지원) |
| **Maintenance LTS** | Node 22 |
| **학습용 권장** | **Node 24 LTS** — 안정적이고 자료가 가장 많다 |

### 릴리스 주기가 바뀐다 (알아둘 것)

Node.js는 2026-10부터 **연 1회 메이저 릴리스**로 전환한다.

```
- 매년 4월 메이저 1회, 10월 LTS 승격
- 모든 릴리스가 LTS가 된다 (홀수 버전 개념 소멸)
- 버전 번호가 연도와 맞춰진다: 27.0.0 = 2027년, 28.0.0 = 2028년
- Node 26이 구 모델의 마지막, Node 27이 새 모델의 첫 번째
```

**학습에 미치는 영향**: 거의 없다. 다만 2027년 이후 자료를 볼 때
"홀수는 불안정 버전"이라는 옛 상식이 더 이상 안 통한다는 것만 기억한다.

## TypeScript 실행 — 빌드 없이 된다

**최근 백엔드 학습 진입장벽이 크게 낮아진 지점.**

```bash
node index.ts     # 그냥 실행된다
```

| 항목 | 상태 |
|---|---|
| 방식 | **타입 스트리핑** — 타입 주석을 공백으로 치환 (SWC 기반 `amaro`) |
| 안정화 | v24.12.0 / v25.2.0에서 stable |
| 플래그 | 불필요 (Node 22.18+ 부터 기본 동작) |
| 소스맵 | 줄 번호가 유지되어 스택트레이스가 정확함 |

### ⚠️ 한계 — 이걸 모르면 헤맨다

```
❌ 타입 검사를 하지 않는다   → 타입을 "지우기만" 한다
   → 검사는 에디터 또는 `tsc --noEmit` 이 담당한다
   → 즉 타입 오류가 있어도 그냥 실행된다

❌ tsconfig.json 을 읽지 않는다
   → paths 별칭(@/utils 등) 안 됨
   → 최신 문법 → 구버전 변환(다운레벨링) 안 됨

❌ "지우기만 해서는 안 되는" 문법에 제약
   → enum, namespace, 파라미터 프로퍼티 등
   → 학습 단계에서는 이것들을 안 쓰면 그만이다
```

> **학습 전략**: 처음에는 타입을 가볍게 쓰고 `node index.ts`로 바로 실행한다.
> 빌드 도구·tsconfig 튜닝에 시간을 쓰지 않는다. **그건 학습이 아니라 세팅이다.**
> 타입 검사가 필요해지면 그때 `tsc --noEmit`을 붙인다.

## 런타임 — Node vs Bun vs Deno

| | Node.js | Bun | Deno |
|---|---|---|---|
| 전문 개발자 채택률 | **~42.7%** | ~2.4% | 소수 |
| 강점 | 생태계·안정성 | 속도(기동 3배, 다수 벤치 3~5배) | TS 우선, 보안 기본, 엔터프라이즈 |
| 약점 | 상대적으로 느림 | **장기 실행 안정성** (메모리 누수 보고 지속) | 생태계 |
| 프로덕션 | 안전한 기본값 | 케이스별 (스테이징 검증 필요) | Node 호환층 성숙 |

> **학습 결론: Node.js를 쓴다.** 논쟁의 여지가 없다.
> 자료의 양, 채용 시장, 라이브러리 호환성 모두 압도적이고,
> **여기서 배운 개념은 Bun/Deno로 거의 그대로 이식된다.**
> Bun은 "빠른 대안이 있다" 정도만 알아두고 6개월 뒤에 다시 본다.

## 웹 프레임워크

| | Express | Fastify | Hono |
|---|---|---|---|
| 주간 다운로드 | ~3,000~3,500만 (1위) | 중간 | ~120만 (18개월 새 6배↑) |
| 성능 | 낮음 | ~62,400 req/s | **~78,200 req/s** |
| TypeScript | 네이티브 아님 | 네이티브 | 네이티브 |
| 크기 | 보통 | 보통 | ~14kB |
| 런타임 | Node만 | Node만 | **Node·Bun·Deno·Cloudflare Workers** |
| 스키마 검증 | 별도 | **내장** | 미들웨어 |

### 학습용 선택

```
1순위: Fastify   — Node 서버 학습에 가장 균형적
                  스키마 검증 내장 = Stage 4 검증 학습과 직결
                  플러그인 생태계 성숙, 실무 채택 많음

대안:  Hono     — 더 가볍고 빠르며 어디서나 돎
                  Express와 API가 비슷해 배우기 쉬움
                  엣지/서버리스에 관심이 생기면 이쪽

참고:  Express  — 자료가 압도적으로 많아 "읽을 일"은 계속 있다
                  다만 신규 학습 주력으로는 권하지 않음
```

> **어느 쪽을 골라도 Stage 1~3에서 배운 개념은 그대로다.**
> 프레임워크 선택에 30분 이상 쓰지 않는다. 이 문서의 1순위를 그냥 따르면 된다.

## 데이터베이스 접근

| | Prisma | Drizzle |
|---|---|---|
| 철학 | 스키마 우선, SQL 추상화 | 코드 우선, **SQL에 가까움** |
| 다운로드 | 2026-05 3주차에 Drizzle에 추월당함 | **2026-05 npm 다운로드 1위 등극** |
| 성능 | Prisma 7(2025-11)에서 Rust 엔진 → TS/WASM 전환, 격차 축소 | 여전히 오버헤드 최소 |
| 도구 | 마이그레이션 성숙, Prisma Studio | 가벼움, 콜드스타트 빠름 |
| 기업 채택 | 강함 (Vercel 배포 21만+ 기업) | 빠르게 성장 |

### 학습용 선택

```
권장: Drizzle  — SQL 구조가 코드에 그대로 보인다
                → 학습 목표가 "SQL을 이해하는 것"이므로 이쪽이 유리
                → 무슨 쿼리가 나가는지 항상 알 수 있다

대안: Prisma   — 추상화가 높아 빠르게 만들 수 있지만
                학습 단계에서는 SQL을 가려버려 역효과 가능
```

> **핵심 원칙은 도구와 무관하다**: 어느 쪽을 쓰든 **생 SQL을 읽고 쓸 줄 아는 상태**를 유지한다.
> ORM만 아는 상태는 성능 문제를 볼 수 없는 상태다. [05-Stage 3 — 데이터베이스와 SQL](05-stage3-데이터베이스와-SQL.md) 참조.

### DB 자체

| 용도 | 선택 |
|---|---|
| **학습·실습** | **SQLite** — 파일 하나, 세팅 0, 트랜잭션 학습에 충분 |
| **실제 서비스** | **PostgreSQL** — 잘 모르겠으면 이것 |

## 인증 — 현재 권장 구성

> ⚠️ **이 영역은 낡은 자료가 특히 많다.** 아래가 현재 권장이다.

```
현재의 표준 조합:
  액세스 토큰   → 메모리 보관, 짧은 수명(5~15분)
  리프레시 토큰 → HttpOnly + Secure + SameSite 쿠키, 서버에 저장

또는 (더 단순하고 대부분의 경우 충분):
  서버 세션 + HttpOnly 쿠키
```

**중요한 현황 3가지:**

1. **"JWT가 세션보다 현대적"은 사실이 아니다.**
   Auth.js(next-auth), Better Auth, Supabase Auth 등 주요 프레임워크의
   기본값이 오히려 **서버 세션**인 경우가 많다.
2. **쿠키 vs JWT는 양자택일이 아니다.** 쿠키는 운반 수단, JWT는 토큰 형식이다. 함께 쓴다.
3. **즉시 로그아웃이 중요한 도메인**(금융·의료·개인 기록)에서는 세션이 명확히 유리하다.

자세한 내용: [07-Stage 5 — 인증과 보안](07-stage5-인증과-보안.md)
