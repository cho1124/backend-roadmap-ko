## 1주차 — 주소는 어떻게 서버에 닿는가

### 계층 모델 (딱 이만큼만)

```
[ HTTP    ]  ← 무슨 말을 할지 (의미)          ★ 백엔드가 사는 층
[ TLS     ]  ← 그 말을 남이 못 보게 (암호화)
[ TCP     ]  ← 말이 순서대로 빠짐없이 도착하게
[ IP      ]  ← 어느 컴퓨터로 보낼지 (주소)
[ 이더넷   ]  ← 실제 전선/전파
```

아래 층은 위 층에게 서비스를 제공한다. HTTP를 다룰 때 TCP 내부는 몰라도 된다.
**단, 무엇을 보장해주는지는 알아야 한다.**

| 계층 | 보장해주는 것 | 백엔드에서 체감되는 순간 |
|---|---|---|
| IP | 목적지까지 어떻게든 감 (유실 가능) | — |
| TCP | 순서 보장, 유실 시 재전송, 연결 개념 | 커넥션 풀, Keep-Alive, 타임아웃 |
| TLS | 도청·변조 방지, 서버 신원 확인 | HTTPS 인증서 만료 사고 |
| HTTP | 요청/응답이라는 의미 구조 | 전부 |

### 이름 → 주소: DNS

`example.com`은 주소가 아니라 **이름**이다. 실제 통신에는 IP가 필요하다.

```
브라우저 캐시 → OS 캐시 → 라우터 → ISP DNS → 루트 → .com → example.com 네임서버
                                                              ↓
                                                        93.184.216.34
```

**왜 알아야 하나**: 서버 장애의 상당 부분이 DNS다.
"서버는 멀쩡한데 접속이 안 됨"의 1순위 용의자. 그리고 **DNS는 캐시된다** —
배포 후 도메인을 바꿔도 즉시 반영되지 않는 이유(TTL)가 여기 있다.

### 포트: 한 컴퓨터 안의 여러 프로그램

IP가 건물 주소면 포트는 호실이다.

| 포트 | 용도 |
|---|---|
| 80 | HTTP |
| 443 | HTTPS |
| 3000, 8080 | 개발용 관례 |
| 5432 / 3306 / 6379 | PostgreSQL / MySQL / Redis |

`localhost:3000`에서 3000이 그것이다. **한 포트는 한 프로세스만** 점유한다 —
`EADDRINUSE` 에러의 정체다.

### 🔧 마이크로 실습 (30분)

```bash
# 1. 응답 헤더만 보기
curl -I https://example.com

# 2. 요청/응답 전 과정 보기 (-v = verbose)
curl -v https://example.com

# 3. 이름이 어떤 IP로 풀리는지
nslookup example.com
```

`curl -v` 출력에서 다음을 눈으로 찾는다:
- `*` 로 시작하는 줄 = 연결/TLS 과정
- `>` 로 시작하는 줄 = **내가 보낸 요청**
- `<` 로 시작하는 줄 = **서버가 준 응답**

이 세 종류를 구분하는 순간 HTTP가 추상적인 게 아니라 **텍스트**라는 게 보인다.

## 2주차 — HTTP 메시지의 정체

### HTTP는 그냥 텍스트다

가장 중요한 깨달음. 브라우저와 서버가 주고받는 건 **약속된 형식의 문자열**이다.

**요청:**
```http
GET /users/42 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbGc...

(본문 — GET은 보통 비어 있음)
```

**응답:**
```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 42

{"id":42,"name":"Alice","role":"admin"}
```

구조는 언제나 동일하다:
```
시작줄 (메서드+경로 / 상태코드)
헤더들 (Key: Value)
빈 줄            ← 헤더와 본문의 경계
본문 (선택)
```

텍스트라서 느리지만, 그 대가로 **어디서든 눈으로 디버깅할 수 있다.**
이 트레이드오프가 HTTP가 30년간 살아남은 이유다.

### 메서드 — 의도의 선언

| 메서드 | 의미 | 안전<sup>1</sup> | 멱등<sup>2</sup> |
|---|---|---|---|
| `GET` | 읽기 | ✅ | ✅ |
| `POST` | 생성 / 그 외 처리 | ❌ | ❌ |
| `PUT` | 통째로 교체 | ❌ | ✅ |
| `PATCH` | 부분 수정 | ❌ | ❌(보통) |
| `DELETE` | 삭제 | ❌ | ✅ |

<sup>1</sup> 안전 = 서버 상태를 바꾸지 않음 · <sup>2</sup> 멱등 = N번 호출해도 1번과 결과 동일

**GET과 POST의 진짜 차이는 파라미터 위치가 아니다.**
- GET은 **안전**하다 → 브라우저가 미리 불러도 되고, 캐시해도 되고, 새로고침해도 된다
- POST는 안전하지 않다 → 새로고침 시 "재전송하시겠습니까?"가 뜨는 이유

GET으로 데이터를 삭제하는 API를 만들면, 크롤러가 사이트를 지나가며 **전부 지운다.**
실제로 여러 번 있었던 사고다.

### 상태 코드 — 결과의 분류

```
1xx 정보      (거의 안 봄)
2xx 성공      200 OK · 201 Created · 204 No Content
3xx 리다이렉션 301 영구이동 · 302 임시 · 304 안 바뀜(캐시 써라)
4xx 클라이언트 잘못  400 · 401 · 403 · 404 · 409 · 422 · 429
5xx 서버 잘못       500 · 502 · 503 · 504
```

**4xx와 5xx의 경계가 핵심이다.** 이건 단순 분류가 아니라 **책임의 선언**이다.

> 4xx = "네 요청이 잘못됐다. 그대로 다시 보내도 똑같다."
> 5xx = "내 잘못이다. 잠시 후 다시 보내면 될 수도 있다."

이 구분 때문에 클라이언트는 **재시도 여부를 자동으로 판단**할 수 있다.
서버 버그로 500을 던져야 할 상황에 400을 주면, 클라이언트는 재시도를 포기한다.

**헷갈리는 짝들:**

| 짝 | 차이 |
|---|---|
| **401 vs 403** | 401 = 누군지 모르겠다(로그인해라) / 403 = 누군지 알겠는데 권한이 없다 |
| **400 vs 422** | 400 = 형식이 깨짐(JSON 파싱 실패) / 422 = 형식은 맞는데 의미가 틀림(나이 -5) |
| **404 vs 403** | 존재 자체를 숨기고 싶으면 403 대신 404를 준다 (정보 유출 방지) |
| **502 vs 503** | 502 = 뒤쪽 서버가 이상한 응답 / 503 = 지금 감당 못 함(과부하·점검) |

### 🔧 마이크로 실습 (1시간) — HTTP를 손으로 쓰기

프레임워크는커녕 `http` 모듈도 안 쓴다. **TCP 소켓에 직접 문자열을 쓴다.**

```ts
// raw-request.ts
import net from 'node:net';

const socket = net.connect(80, 'example.com', () => {
  socket.write(
    'GET / HTTP/1.1\r\n' +
    'Host: example.com\r\n' +
    'Connection: close\r\n' +
    '\r\n'                     // ← 이 빈 줄이 "헤더 끝" 신호
  );
});

socket.on('data', (chunk) => process.stdout.write(chunk.toString()));
```

```bash
node raw-request.ts
```

**이 실습의 목적**: HTTP가 마법이 아니라 **약속된 문자열**임을 손으로 확인하는 것.
`\r\n`을 하나 빼먹으면 서버가 응답을 안 준다 — 그 순간 "아, 진짜 그냥 규약이구나"가 온다.

## 3주차 — 상태 없는 프로토콜에 상태 얹기

### HTTP는 기억하지 못한다

HTTP는 **stateless**다. 요청 하나하나가 완전히 독립이고, 서버는 이전 요청을 기억하지 않는다.

그런데 로그인은 유지된다. 어떻게?
→ **매 요청마다 신분증을 다시 보여주기 때문이다.** 그 신분증이 쿠키/토큰이다.

### 쿠키

```http
# 서버 → 클라이언트 (응답)
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600

# 클라이언트 → 서버 (이후 모든 요청에 자동 첨부)
Cookie: sessionId=abc123
```

**보안 속성 3개 — 각각 다른 공격을 막는다:**

| 속성 | 막는 것 | 설명 |
|---|---|---|
| `HttpOnly` | **XSS로 쿠키 훔치기** | JS에서 `document.cookie`로 못 읽음 |
| `Secure` | **평문 도청** | HTTPS에서만 전송 |
| `SameSite` | **CSRF** | 다른 사이트에서 온 요청엔 안 붙임 |

이 3개는 [07-Stage 5 — 인증과 보안](07-stage5-인증과-보안.md)에서 다시 나온다.
지금은 **"각각 다른 걸 막는다"**만 확실히 해둔다.

### 캐시 — 안 보내는 게 제일 빠르다

```http
Cache-Control: max-age=3600        # 1시간 동안 재요청 하지 마라
ETag: "v3-a9f2"                    # 이 응답의 지문
If-None-Match: "v3-a9f2"           # (재요청 시) 이 지문이면 그냥 304 줘
→ HTTP/1.1 304 Not Modified        # 본문 없음. 네 캐시 써라
```

**가장 빠른 응답은 안 보내는 것이다.** 304는 본문이 0바이트다.
서버 최적화에서 캐시가 항상 상위권에 있는 이유가 이것이다.

### CORS — 가장 많이 욕먹고 가장 많이 오해받는 것

**증상**: 브라우저 콘솔에 빨간 CORS 에러. 그런데 `curl`로는 잘 된다.

**이유**: CORS는 **브라우저가 스스로에게 거는 제약**이다. 서버 기능이 아니다.

```
왜 있나: 브라우저는 로그인 쿠키를 자동으로 붙인다.
        그래서 evil.com 의 스크립트가 bank.com 에 요청을 보내면
        은행 쿠키가 딸려간다. → 아무 사이트나 남의 은행 API를 부를 수 있다.

그래서: 브라우저가 "다른 출처로 보내는 요청"의 응답을
        서버가 명시적으로 허락(Access-Control-Allow-Origin)하지 않으면
        스크립트에게 넘겨주지 않는다.
```

**핵심 정정 3가지** (인터넷 자료가 자주 틀리는 부분):
1. CORS는 서버를 보호하는 게 아니라 **사용자를 보호**한다
2. 요청은 실제로 **서버에 도달했다**. 브라우저가 응답을 감췄을 뿐이다
   → 그래서 CORS 에러가 나도 서버에서는 데이터가 이미 바뀌었을 수 있다
3. `curl`·서버 간 통신·모바일 앱에는 CORS가 **없다**

### HTTPS / TLS (개념만)

```
1. 클라이언트: "안녕, 나 이런 암호화 방식들 지원해"
2. 서버: "이걸로 하자. 여기 내 인증서(신원증명)"
3. 클라이언트: 인증서가 신뢰할 만한 기관(CA) 서명인지 검증
4. 양쪽: 이번 연결에만 쓸 대칭키를 안전하게 합의
5. 이후: 그 키로 빠르게 암호화 통신
```

알아야 할 실무 사실:
- 인증서는 **만료된다** (Let's Encrypt는 90일) → 자동 갱신이 안 되면 서비스가 죽는다
- HTTPS는 **도청·변조·위장**을 막지만, **서버 코드의 취약점은 못 막는다**
- 보통 앱 서버가 아니라 **리버스 프록시가 TLS를 끝낸다** (Stage 6)

### 🔧 마이크로 실습 (30분)

브라우저 개발자도구 → Network 탭에서 아무 사이트나 열고:
- [ ] 요청 하나 클릭 → Headers 탭에서 요청/응답 헤더 전부 읽기
- [ ] Status가 `304`인 요청 찾기 → 크기(Size)가 왜 작은지 확인
- [ ] Application 탭 → Cookies에서 `HttpOnly` 체크된 쿠키 찾기
- [ ] 새로고침 vs 강력 새로고침(Ctrl+Shift+R)의 요청 수 차이 관찰

## 4주차 — 버전, 연결, 그리고 첫 서버

### HTTP/1.1 → 2 → 3

| 버전 | 핵심 변화 | 해결한 문제 |
|---|---|---|
| **1.1** | Keep-Alive (연결 재사용) | 요청마다 TCP 연결 맺는 낭비 |
| **2** | 멀티플렉싱, 헤더 압축, 바이너리 | HOL 블로킹(한 요청이 뒤를 막음) |
| **3** | TCP → QUIC(UDP 기반) | TCP 레벨 HOL, 연결 전환(모바일) |

**교양 수준에서 알아야 할 것**: 셋 다 **의미(메서드·상태코드·헤더)는 동일**하다.
바뀐 건 **전송 방식**뿐이다. 그래서 애플리케이션 코드는 대개 그대로다.

> 실무 감각: HTTP/2·3은 대부분 **리버스 프록시가 처리**하고,
> 뒤쪽 앱 서버는 1.1로 받는 구성이 흔하다.

### Keep-Alive와 커넥션 풀

TCP 연결을 맺는 데는 왕복 시간(RTT)이 든다. TLS까지면 더 든다.
그래서 **한 번 맺은 연결을 재사용**한다.

이건 "비싼 자원을 만들지 말고 빌려 쓴다"는 **풀링(pooling)** 패턴이고,
DB 커넥션 풀(Stage 3)도 정확히 같은 이유로 존재한다.

### 🔧 마이크로 실습 (1시간) — 첫 HTTP 서버

이제 `node:http`를 쓴다. **프레임워크는 아직 없다.**

```ts
// server.ts
import http from 'node:http';

const server = http.createServer((req, res) => {
  // req: 들어온 요청  /  res: 내가 쓸 응답
  console.log(`${req.method} ${req.url}`);

  if (req.method === 'GET' && req.url === '/hello') {
    res.writeHead(200, { 'Content-Type': 'application/json; charset=utf-8' });
    res.end(JSON.stringify({ message: '안녕하세요' }));
    return;
  }

  res.writeHead(404, { 'Content-Type': 'application/json; charset=utf-8' });
  res.end(JSON.stringify({ error: 'Not Found' }));
});

server.listen(3000, () => console.log('http://localhost:3000'));
```

**실습 확장 (여기서 배움이 나온다):**
- [ ] `Content-Type`을 지우면 브라우저가 어떻게 반응하나?
- [ ] `charset=utf-8`을 빼면 한글이 어떻게 되나?
- [ ] `res.end()`를 호출하지 않으면? (브라우저가 계속 기다린다 — **응답을 끝내는 책임은 나에게 있다**)
- [ ] 같은 포트로 서버를 하나 더 띄우면? (`EADDRINUSE`)
- [ ] `/hello?name=alice` 로 요청하면 `req.url`에 뭐가 들어오나?

마지막 항목에서 **쿼리 스트링 파싱이 자동이 아니라는 것**을 알게 된다.
→ 이 불편함들이 쌓여서 Stage 4의 프레임워크로 이어진다. **순서가 이래야 의미가 있다.**

## 결론

> HTTP는 **"텍스트로 된 함수 호출 규약"**이다.
>
> 메서드 = 함수 이름, URL = 대상, 헤더 = 메타데이터, 본문 = 인자, 상태코드 = 반환값 분류.
>
> 이 비유가 몸에 붙으면 나머지 백엔드는 전부 "그 함수 안에서 무엇을 하느냐"의 문제로 단순해진다.
