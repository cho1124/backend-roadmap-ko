# 01 - 백엔드의 5가지 근본 전제

> 허브: [00-백엔드 학습 허브](00-백엔드-학습-허브.md)
> **이 문서를 가장 먼저 읽는다.** 기술이 아니라 사고방식에 관한 문서다.

---

## 핵심 요약

```
대부분의 프로그램 = 한 사람이 쓰는 상태를 다룬다
서버            = 모든 사람의 상태를 한 곳에서 지킨다
```

서버 개발이 어려운 이유는 **모르는 기술** 때문이 아니라,
평소 프로그래밍에서 당연하게 여기던 전제 5개가 **전부 뒤집히기** 때문이다.
이 5개를 먼저 짚는다. 나머지 문서는 전부 여기서 파생된다.

---

## ① 동시성이 예외가 아니라 기본값이다

일반적인 프로그램에서 함수는 **한 번에 하나씩** 실행된다.
그래서 전역 변수에 값을 담아두고 나중에 꺼내 써도 대체로 안전하다.

서버에서는 **같은 함수가 지금 이 순간 수백 개 동시에 실행 중**이다.

```ts
// ❌ 서버에서는 폭탄이 되는 코드
let currentUser = null;              // 모듈 전역

async function handleLogin(req) {
  currentUser = req.body.user;       // A 요청이 여기까지 왔을 때
  const data = await loadData();     // ← 대기 중에 B 요청이 currentUser를 덮어씀
  return render(currentUser);        // A가 B의 유저 정보를 받는다
}
```

이 버그는 **혼자 테스트하면 절대 안 나온다.** 사용자가 둘 이상일 때만 터진다.
그래서 개발 중에는 멀쩡하다가 배포 후에 재현이 안 되는 형태로 나타난다.

> **제1계율**: 요청 사이에 공유되는 가변 상태를 만들지 않는다 (**stateless**).
> 상태가 필요하면 프로세스 밖(DB, 캐시)에 둔다.

이 한 줄이 [05-Stage 3 — 데이터베이스와 SQL](05-stage3-데이터베이스와-SQL.md)의 존재 이유이고,
[08-Stage 6 — 배포와 운영](08-stage6-배포와-운영.md)에서 서버를 여러 대로 늘릴 수 있는 조건이다.

---

## ② 입력은 전부 적대적이다

프로그램 내부의 함수 호출은 내가 통제한다. 인자 검증은 편의 기능이다.

서버로 들어오는 요청은 **누가 무엇을 보냈는지 아무 보장이 없다.**
클라이언트(웹 페이지, 모바일 앱, CLI 도구)의 코드는 **전부 상대방 기기에 있다** —
즉 상대가 마음대로 뜯어보고 고칠 수 있다.

```
클라이언트에서 "레벨 10 이상만 버튼 활성화"  → 아무 의미 없음
                                            (버튼을 안 눌러도 요청은 보낼 수 있다)
서버에서 "레벨 10 이상만 처리"              → 이것만 실제 규칙이다
```

> **계율**: 클라이언트의 검증은 UX이고, 서버의 검증은 보안이다. 둘은 다른 일을 한다.
> 클라이언트에서 검증했다는 이유로 서버 검증을 생략하는 순간 구멍이 뚫린다.

이것이 **신뢰 경계(trust boundary)** 개념이고, [07-Stage 5 — 인증과 보안](07-stage5-인증과-보안.md)의 토대다.

---

## ③ 실패가 예외가 아니라 정상 경로다

같은 프로세스 안의 함수 호출은 실패하지 않는다. 호출하면 실행된다.

네트워크 너머의 호출은 **정상적으로 실패한다.**

- 요청이 도달했는데 응답이 유실될 수 있다 (성공했는지 알 수 없음)
- 같은 요청이 두 번 도착할 수 있다 (재시도 때문에)
- 상대 서버가 응답을 3초 뒤에 줄 수 있다
- DB가 순간적으로 연결을 거부할 수 있다

그래서 백엔드에는 일반 프로그래밍에 없는 개념들이 생긴다.

| 개념 | 뜻 | 왜 필요한가 |
|---|---|---|
| **멱등성(idempotency)** | 같은 요청을 N번 보내도 결과가 1번과 같음 | 재시도가 결제를 3번 하면 안 되니까 |
| **타임아웃** | 언제까지 기다릴지 미리 정함 | 안 정하면 영원히 기다리다 서버가 막힘 |
| **재시도 + 백오프** | 실패 시 간격을 늘리며 다시 시도 | 즉시 재시도는 죽어가는 서버를 확실히 죽임 |
| **서킷 브레이커** | 계속 실패하면 아예 시도를 멈춤 | 장애 전파 차단 |

**핵심**: 이것들은 "안정성을 높이는 추가 기능"이 아니라 **기본 설계 조건**이다.
나중에 붙이려고 하면 구조를 갈아엎어야 한다.

---

## ④ 화면이 없다 — 관측되지 않으면 존재하지 않는다

로컬에서 개발할 때는 디버깅 수단이 풍부하다.
브레이크포인트를 걸고, 변수를 들여다보고, 출력을 눈으로 본다.

운영 중인 서버는 **아무것도 안 보인다.** 어딘가에서 혼자 돌고 있다.
잘 돌고 있는지 아는 유일한 방법은 **서버가 스스로 남긴 기록**뿐이다.

```
로그(Log)      — 무슨 일이 있었나 (사건 기록)
메트릭(Metric)  — 얼마나 자주/빠른가 (숫자 시계열)
트레이스(Trace) — 한 요청이 어디를 거쳤나 (경로 추적)
```

이 셋을 **관측 가능성(Observability)**이라 하고, [08-Stage 6 — 배포와 운영](08-stage6-배포와-운영.md)에서 다룬다.

> **계율**: 로그를 안 남긴 코드 경로는, 문제가 생겨도 영원히 원인을 알 수 없다.

그래서 로그를 심는 것은 부가 작업이 아니라 **서버에 감각기관을 달아주는 일**이다.

---

## ⑤ 프로그램이 끝나지 않는다

일반 프로그램은 켜고 끈다. 메모리가 조금 새도 종료하면 정리된다.

서버는 **몇 달을 연속으로 돈다.** 그래서 평소 무시해도 됐던 것들이 치명적이 된다.

| 일반 프로그램에서 | 서버에서 |
|---|---|
| 작은 메모리 누수 → 종료하면 됨 | 작은 누수 → 2주 뒤 OOM으로 서비스 다운 |
| 커넥션 안 닫음 → 별일 없음 | 안 닫음 → 커넥션 풀 고갈로 전면 정지 |
| 시작할 때 무거운 초기화 → 로딩 화면 | 배포마다 발생 → 무중단 배포 설계 필요 |
| 크래시 → 나만 재시작 | 크래시 → **전원 서비스 중단** |

> **계율**: 서버에서 "열었으면 닫는다"는 스타일이 아니라 생존 조건이다.

그리고 마지막 줄이 서버 개발의 심리적 무게를 만든다.
**내 실수의 영향 범위가 나 혼자가 아니다.**

---

## 서버는 결국 무한 루프다

다섯 전제를 하나로 묶은 그림. 여기서 모든 게 파생된다.

```ts
while (true) {
  const request  = await 소켓에서_바이트를_읽는다();   // ← Stage 1 (HTTP)
  const parsed   = 바이트를_의미로_해석한다(request);   // ← Stage 1
  // 위 await 동안 다른 요청도 처리된다                 // ← Stage 2 (이벤트 루프)
  const who      = 이_사람이_누군지_확인한다(parsed);   // ← Stage 5 (인증)
  const data     = await 상태를_읽거나_쓴다(who);       // ← Stage 3 (DB)
  const response = 결과를_약속된_모양으로_만든다(data); // ← Stage 4 (API 설계)
  소켓에_바이트를_쓴다(response);
  로그를_남긴다();                                      // ← Stage 6 (운영)
}
```

**로드맵의 6개 스테이지는 이 루프의 각 줄을 하나씩 파는 것이다.**
어느 스테이지를 공부하다 길을 잃으면 이 그림으로 돌아온다.

## "백엔드"라는 말이 가리키는 4가지

용어가 뭉뚱그려져 있어서 학습 방향을 잃기 쉬운 지점. 미리 갈라둔다.

| 영역 | 하는 일 | 이 로드맵의 커버 |
|---|---|---|
| **애플리케이션 백엔드** | API 서버, 비즈니스 로직, DB 접근 | ★ 전부 (Stage 1~5) |
| **데이터 엔지니어링** | 파이프라인, ETL, 데이터 웨어하우스 | ✗ |
| **인프라 / SRE** | 서버 운영, 오케스트레이션, 모니터링 | 기초만 (Stage 6) |
| **플랫폼 / 분산 시스템** | 합의 알고리즘, 샤딩, 대규모 트래픽 | ✗ (6개월 뒤 선택) |

6개월로 도달 가능한 것은 **애플리케이션 백엔드의 탄탄한 기반**이다.
쿠버네티스와 카프카는 이 로드맵의 목적지가 아니고, 그래도 된다.
**1번을 모른 채 3·4번을 배우면 아무것도 남지 않는다.**

---

## 자가 점검

이 문서를 덮고 답할 수 있으면 통과다.

1. 서버에서 모듈 전역 변수에 상태를 넣으면 왜 위험한가?
2. 그 버그가 개발 중에는 왜 재현되지 않는가?
3. 클라이언트에서 이미 검증했는데 서버에서 또 검증하는 이유는?
4. 멱등성이 필요한 이유를 결제 예시로 설명해보라
5. 로그·메트릭·트레이스는 각각 어떤 질문에 답하는가?
6. 메모리 누수가 서버에서 특히 치명적인 이유는?

---

## 결론

> 백엔드로 넘어가는 건 **새 언어를 배우는 일이 아니라, 전제를 바꾸는 일**이다.
>
> "하나씩 실행, 신뢰할 수 있는 입력, 성공하는 호출, 보이는 상태, 끝나는 프로그램"에서
> "동시 실행, 적대적 입력, 실패하는 호출, 보이지 않는 상태, 끝나지 않는 프로그램"으로.
>
> 문법은 일주일이면 되지만, 이 전제 전환에는 몇 달이 걸린다. 그게 이 로드맵의 6개월이다.
