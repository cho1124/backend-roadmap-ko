# 백엔드 학습 허브

> 대상: 프로그래밍 경험은 있지만 서버 개발은 처음인 사람
> 주력 스택: **Node.js + TypeScript**
> 목표: **원리를 체계적으로 이해하기** (특정 프레임워크 숙달이 아님)
> 분량: 주 5~7시간 기준 **6개월 / 24주 / 약 164시간**

---

## 이 문서 세트가 하려는 것

백엔드는 "배울 게 많은" 분야가 아니라 **"배울 게 흩어져 있는"** 분야다.
프론트엔드는 화면이라는 결과물이 학습을 붙잡아 주지만, 백엔드는 눈에 보이는 게 없어서
HTTP·DB·인증·배포가 서로 무관한 파편처럼 느껴진다.

이 문서 세트는 그 파편들을 **하나의 인과 사슬**로 엮는다.

```
요청이 들어온다 (HTTP)
  → 서버가 그걸 처리한다 (런타임/동시성)
    → 상태는 밖에 저장한다 (DB)
      → 그 처리를 계약으로 노출한다 (API 설계)
        → 아무나 못 하게 막는다 (인증/보안)
          → 계속 살아있게 만든다 (배포/운영)
```

이 순서가 곧 로드맵이다. 각 단계는 **앞 단계에서 생긴 문제를 해결하려고 태어난 기술**이다.
그 인과를 이해하면 프레임워크는 나중에 문서만 읽어도 된다.

---

## 학습 가이드라인 6원칙

### 1. 프레임워크를 먼저 배우지 않는다

가장 흔한 실패 경로다.

```
❌ Express 튜토리얼 → API 만들어짐 → HTTP를 모름 → 뭐가 잘못돼도 원인 파악 불가
✅ node:http 로 직접 → 불편함 → "그래서 프레임워크가 있구나" → Express/Fastify
```

프레임워크는 **불편함을 겪은 사람에게만 의미가 있다.**
`app.get('/users', handler)` 한 줄이 무엇을 감추고 있는지 모르면, 그건 지식이 아니라 주문(呪文)이다.

이 로드맵에서 프레임워크는 **Stage 4에 가서야** 등장한다. 의도된 순서다.

### 2. "왜 이렇게 됐는가"를 항상 먼저 묻는다

백엔드 기술은 대부분 **제약의 역사**다. 기능이 아니라 상처의 흔적이다.

- REST는 왜 생겼나 → 분산 시스템에서 캐시가 안 먹혀서
- JWT는 왜 생겼나 → 서버가 여러 대라 세션을 공유하기 어려워서
- 컨테이너는 왜 생겼나 → "내 PC에선 되는데요"를 없애려고
- ORM은 왜 생겼나 → SQL 문자열 조립이 타입 안전하지 않아서
- 그런데 왜 ORM을 다시 걷어내나 → 추상화가 SQL을 숨겨서 성능을 못 봐서

원리 이해가 목표라면 **이 "왜"가 본체이고, 문법은 부산물**이다.

### 3. 오래 남을 것부터 배운다 (감가상각 순서)

| 지식 | 유효기간 | 우선순위 |
|---|---|---|
| HTTP / TCP / DNS | 20년+ | ★★★ |
| SQL / 관계형 모델 | 40년+ (1970년대 이론) | ★★★ |
| 동시성 / 트랜잭션 / 일관성 | 영구 | ★★★ |
| 인증·인가의 원리 | 10년+ | ★★ |
| Linux / 컨테이너 | 10년+ | ★★ |
| 특정 프레임워크 (Express, Nest…) | 3~5년 | ★ |
| 특정 ORM / 라이브러리 | 2~3년 | ☆ |

**시간 예산이 한정적일수록** 이 순서를 지키는 게 결정적이다.
지금 유행하는 프레임워크에 20시간을 쓰면 그 20시간은 3년 뒤 0이 된다.
HTTP에 쓴 20시간은 20년 간다.

### 4. 실습은 "개념 1개당 30줄"

읽기만 하면 안 남는다. 다만 목표가 제품이 아니므로 실습은 작게.

```
❌ "블로그 서비스 만들기" (4주짜리, 개념 20개가 섞여서 뭘 배웠는지 모름)
✅ "쿠키로 로그인 상태 유지하기" (30줄, 개념 1개, 30분)
```

각 스테이지 문서에 이런 마이크로 실습이 들어 있다.
큰 프로젝트는 스테이지 두세 개마다 한 번씩만: [09-실습 프로젝트 3단계](09-실습-프로젝트-3단계.md)

### 5. 완료 기준은 "말로 설명할 수 있는가"

"돌아간다"는 완료 기준이 아니다. 복붙해도 돌아가니까.

> **자가 점검 질문**: 이 개념을 백엔드 안 하는 동료에게 3분 안에 설명할 수 있는가?

각 스테이지 끝에 이 질문 목록이 있다. 막히면 그 스테이지는 안 끝난 것이다.

### 6. 매주 요약 1장을 직접 쓴다

읽은 것과 이해한 것을 가르는 가장 값싼 장치다.
한 주에 배운 것을 **자기 말로** 한 장 정리한다. 못 쓰면 안 배운 것이다.

## 흔한 함정 지도

학습 시간을 통째로 태워먹는 것들. 미리 알고 피한다.

| 함정                 | 증상                                    | 회피법                                   |
| ------------------ | ------------------------------------- | ------------------------------------- |
| **튜토리얼 지옥**        | 강의 3개 완주했는데 빈 파일에서 아무것도 못 씀           | 강의 1개당 실습 3개. 강의 없이 만드는 시간 > 강의 보는 시간 |
| **스택 쇼핑**          | Nest? Hono? Drizzle? Prisma? 고르다 2주 감 | 스택은 이 문서가 이미 정했다. 6개월 뒤 다시 고민         |
| **인프라 조기 진입**      | 4주차에 Kubernetes 영상을 보고 있음             | 인프라는 Stage 6. 그 전엔 손대지 않는다            |
| **프레임워크 = 백엔드 착각** | Express는 아는데 HTTP 상태코드를 모름            | 원칙 1                                  |
| **완벽주의 리팩터링**      | 실습 코드를 클린아키텍처로 3번 갈아엎음                | 실습 코드는 버리는 코드다. 개념만 남기고 삭제            |
| **AI 복붙 학습**       | 코드는 돌아가는데 왜 돌아가는지 모름                  | AI에게 코드 말고 **"왜"를 묻는다**               |

마지막 항목이 지금 기준으로 제일 위험하다.
AI는 백엔드 코드를 매우 잘 쓴다 — 그래서 **학습을 건너뛰기가 그 어느 때보다 쉽다.**
원리 이해가 목표라면 AI는 "코드 생성기"가 아니라 **"왜 이렇게 하냐고 따져 묻는 상대"**로 쓴다.

> **판정 기준**: 생성된 코드를 한 줄씩 설명할 수 있으면 통과.
> 설명 못 하는 줄이 있으면 그건 내 코드가 아니다.
