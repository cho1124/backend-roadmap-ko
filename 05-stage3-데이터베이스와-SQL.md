## 0. 왜 DB인가 — 파일이면 안 되나

```
파일에 저장하면 안 되는 이유:

① 동시 쓰기    두 요청이 같은 파일을 동시에 쓰면 깨진다
② 부분 실패    3줄 중 2줄 쓰고 죽으면? 데이터가 반쯤 망가진 상태로 남는다

④ 서버 여러 대  서버 2대가 각자 파일을 갖고 있으면 진실이 두 개다
```

**①②는 트랜잭션, ③은 인덱스, ④는 중앙 집중**이 답이다.

## 1주차 — 관계형 모델과 스키마 설계

### 테이블은 "타입"이고 행은 "인스턴스"다

```ts
// 코드 관점
type User = { id: number; email: string; name: string };
const users: User[] = [];
```

```sql
-- 같은 것
CREATE TABLE users (
    id    INTEGER PRIMARY KEY,
    email TEXT NOT NULL UNIQUE,
    name  TEXT NOT NULL
);
```

**결정적 차이**: 코드에서 `User`가 `Post[]`를 들고 있듯 **중첩할 수 없다.**
관계형 모델은 평평(flat)하다. 관계는 **키로 가리킨다.**

```sql
CREATE TABLE posts (
    id      INTEGER PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id),   -- ← 외래키: 이게 관계다
    title   TEXT NOT NULL
);
```

### 키 3종

| 종류 | 뜻 |
|---|---|
| **기본키 (PK)** | 이 행을 유일하게 식별. 중복·NULL 불가 |
| **외래키 (FK)** | 다른 테이블의 PK를 가리킴. 관계의 실체 |
| **유니크 키** | PK는 아니지만 중복 불가 (이메일 등) |

### 관계의 3가지 모양

```
1:1   users ↔ user_profiles          (프로필 테이블에 user_id UNIQUE)
1:N   users → posts                  (posts 에 user_id)      ← 가장 흔함
N:M   users ↔ groups                 (중간 테이블 user_groups 필요)
```

**N:M에는 반드시 중간 테이블이 필요하다.** 관계형 모델에는 "배열 컬럼"이 없기 때문이다.
(일부 DB에 배열/JSON 타입이 있긴 하지만, 관계를 그걸로 표현하면 대부분 후회한다.)

### 정규화 — 진실을 한 곳에만 둔다

```sql
-- ❌ 비정규화: 사용자 이름이 여러 곳에 중복
posts(id, user_id, user_name, title)
-- 사용자가 이름을 바꾸면? 모든 posts를 찾아 고쳐야 한다. 하나라도 놓치면 진실이 두 개.

-- ✅ 정규화: 이름은 users에만 있고, posts는 가리키기만
posts(id, user_id, title)
```

> **핵심 원리**: **"하나의 사실은 한 곳에만 기록한다."**
> 중복은 저장 공간 문제가 아니라 **불일치 위험** 문제다.

**단, 정규화가 항상 옳은 건 아니다.** JOIN이 너무 많아지면 의도적으로 중복을 허용한다(비정규화).
교양 수준에서는 **"기본은 정규화, 성능 문제가 실제로 측정되면 비정규화"** 순서만 알면 된다.

### 스키마 설계 절차

```
1. 명사를 뽑는다        → 사용자, 게시글, 태그
2. 각각 테이블로        → users, posts, tags
3. 관계를 정한다        → users 1:N posts, posts N:M tags
4. 각 테이블의 속성     → 컬럼
5. 제약을 건다          → NOT NULL, UNIQUE, FK, CHECK
```

**제약(constraint)을 아끼지 않는다.**
DB 제약은 "코드가 실수해도 데이터는 안 망가진다"는 마지막 방어선이다.
애플리케이션 검증은 빠뜨릴 수 있지만 DB 제약은 안 빠뜨린다.

### 🔧 마이크로 실습 (1시간)

## 2주차 — SQL과 인덱스

### SQL은 "어떻게"가 아니라 "무엇을" 쓴다

```sql
SELECT u.name, COUNT(p.id) AS post_count
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
WHERE u.created_at > '2026-01-01'
GROUP BY u.id, u.name
HAVING COUNT(p.id) > 5
ORDER BY post_count DESC
LIMIT 10;
```

**루프가 없다.** 어떻게 찾을지는 DB의 쿼리 플래너가 정한다.
이것이 **선언형**이고, 그래서 DB가 알아서 최적화할 여지가 생긴다.

### 실행 순서 (헷갈리는 지점)

```
작성 순서: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
실행 순서: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

이래서 `SELECT`에서 붙인 별칭(`post_count`)을 `WHERE`에서는 못 쓰고 `ORDER BY`에서는 쓸 수 있다.

### JOIN 4종

| JOIN | 결과 |
|---|---|
| `INNER JOIN` | 양쪽 다 있는 것만 |
| `LEFT JOIN` | 왼쪽 전부 + 오른쪽 있으면 (없으면 NULL) |
| `RIGHT JOIN` | 반대 (거의 안 씀 — LEFT로 뒤집어 쓴다) |
| `FULL OUTER` | 양쪽 전부 (드묾) |

**실무에서는 INNER와 LEFT가 95%다.**
"게시글 없는 사용자도 포함해야 하나?" → 예면 LEFT, 아니면 INNER.

### 인덱스 — 책의 색인

```sql
CREATE INDEX idx_posts_user_id ON posts(user_id);
```

없으면: 100만 행을 처음부터 끝까지 훑는다 (**Full Scan**)
있으면: 정렬된 구조(B-Tree)에서 바로 찾는다 (**Index Scan**)

**그런데 왜 전부 인덱스를 걸지 않는가:**

| 비용 | 설명 |
|---|---|
| 쓰기가 느려짐 | INSERT/UPDATE마다 인덱스도 갱신해야 한다 |
| 저장공간 | 인덱스도 데이터다 |
| 플래너 혼란 | 너무 많으면 잘못된 인덱스를 고를 수 있다 |

> **핵심**: 인덱스는 **읽기를 빠르게 하려고 쓰기와 공간을 지불하는** 거래다.
> 무조건 좋은 게 아니라 **읽기/쓰기 비율**의 문제다.

**인덱스를 거는 기준**: `WHERE`, `JOIN ON`, `ORDER BY`에 자주 등장하는 컬럼.

### 실행 계획 읽기 — 이게 진짜 도구다

```sql
EXPLAIN QUERY PLAN SELECT * FROM posts WHERE user_id = 42;   -- SQLite
EXPLAIN ANALYZE    SELECT * FROM posts WHERE user_id = 42;   -- PostgreSQL
```

찾아볼 것: `SCAN`(전체 훑기 — 나쁨) vs `SEARCH ... USING INDEX`(인덱스 사용 — 좋음)

**"느린 것 같다"고 추측하지 않고 실제로 무엇을 하는지 본다.**
성능 작업의 기본 태도다.

### 🔧 마이크로 실습 (1.5시간)

```sql
-- 1. 테이블 만들고 행 10만 개 넣기
-- 2. 인덱스 없이 WHERE 조회 → 시간 측정 + EXPLAIN
-- 3. 인덱스 생성
-- 4. 같은 조회 → 시간 측정 + EXPLAIN 비교
```

**숫자가 눈으로 바뀌는 순간이 인덱스를 이해하는 순간이다.**
SQLite면 파일 하나로 끝나서 세팅 비용이 거의 없다 — 이 실습은 SQLite로 한다.

## 3주차 — 트랜잭션과 동시성 ★★★

> **이 스테이지의 심장이다.** 일반 프로그래밍에 대응 개념이 거의 없는 영역이므로
> 시간을 더 써도 된다.

### 문제: 부분 실패

```
계좌 이체:
  1. A에서 1만원 뺀다   ← 성공
  2. B에 1만원 넣는다   ← 여기서 서버가 죽으면?

→ 1만원이 세상에서 사라진다
```

**트랜잭션**: 여러 작업을 **한 덩어리**로 묶어서, 전부 되거나 전부 안 되게 한다.

```sql
BEGIN;
  UPDATE accounts SET balance = balance - 10000 WHERE id = 'A';
  UPDATE accounts SET balance = balance + 10000 WHERE id = 'B';
COMMIT;   -- 여기서 비로소 확정. 중간에 죽으면 ROLLBACK되어 아무 일도 없던 것이 된다
```

### ACID

| 글자 | 뜻 | 한 줄 |
|---|---|---|
| **A**tomicity | 원자성 | 전부 되거나 전부 안 되거나 |
| **C**onsistency | 일관성 | 제약을 깨는 상태로는 끝나지 않음 |
| **I**solation | 격리성 | 동시에 돌아도 혼자 돈 것처럼 보임 |
| **D**urability | 지속성 | COMMIT했으면 정전에도 남음 |

**I(격리성)가 제일 어렵고 제일 중요하다.**

### 동시성이 만드는 이상 현상

```
갱신 유실 (Lost Update)
  T1: 재고 읽음 (10)
  T2: 재고 읽음 (10)
  T1: 10-1=9 저장
  T2: 10-1=9 저장     ← 2개 팔렸는데 재고는 1만 줄었다
```

```
더티 리드 (Dirty Read)      아직 커밋 안 된 남의 데이터를 읽음
반복 불가능 읽기            같은 걸 두 번 읽었는데 값이 다름
팬텀 리드                  같은 조건으로 두 번 조회했는데 행 개수가 다름
```

### 격리 수준 — 안전성과 속도의 거래

| 수준 | 더티리드 | 반복불가 | 팬텀 | 비고 |
|---|---|---|---|---|
| READ UNCOMMITTED | 발생 | 발생 | 발생 | 거의 안 씀 |
| READ COMMITTED | 방지 | 발생 | 발생 | PostgreSQL 기본 |
| REPEATABLE READ | 방지 | 방지 | 발생* | MySQL 기본 |
| SERIALIZABLE | 방지 | 방지 | 방지 | 가장 안전, 가장 느림 |

> **핵심 통찰**: 격리 수준이란 **"얼마나 안전하게 할지를 성능으로 사는" 다이얼**이다.
> 기본값은 대개 READ COMMITTED — 즉 **기본 설정은 완벽하지 않다.**
> 재고·잔액·좌석처럼 정확성이 중요한 곳은 명시적으로 올리거나 락을 건다.

### 락 — 낙관 vs 비관

```sql
-- 비관적 락: 미리 잠근다. "충돌이 날 거라 가정"
SELECT * FROM items WHERE id = 1 FOR UPDATE;

-- 낙관적 락: 안 잠그고, 저장할 때 바뀌었는지 확인. "충돌이 드물 거라 가정"
UPDATE items SET stock = 9, version = 6 WHERE id = 1 AND version = 5;
-- 영향받은 행이 0이면 → 누가 먼저 바꿨다 → 재시도
```

**데드락**: 서로가 서로를 기다림. DB가 감지해서 한쪽을 죽인다.
→ 그래서 **애플리케이션은 트랜잭션 실패 시 재시도할 수 있어야 한다.**

### 실무 규칙 4개

```
① 트랜잭션은 짧게       — 길면 락을 오래 잡아 전체가 느려진다
② 트랜잭션 안에서 외부 API 호출 금지  — 상대가 3초 걸리면 락을 3초 잡는다
③ 순서를 일정하게       — 항상 id 오름차순으로 잠그면 데드락이 줄어든다
④ 재시도 가능하게 짠다  — 데드락/충돌은 정상 경로다
```

②는 실수하기 매우 쉽고 매우 치명적이다.

### 🔧 마이크로 실습 (1.5시간) — 갱신 유실 재현

```ts
// 두 개의 커넥션으로 동시에 같은 행을 UPDATE
// 1. 트랜잭션 없이 → 갱신 유실 재현
// 2. 트랜잭션 + 적절한 락 → 방지 확인
```

**직접 데이터를 잃어봐야 트랜잭션이 몸에 들어온다.**
개념으로만 읽으면 "당연한 얘기"로 흘러가고 실무에서 그대로 사고를 낸다.

## 4주차 — ORM, N+1, 마이그레이션

### 접근 방식 3층

```
① 생 SQL      최대 제어, 타입 안전성 없음, 문자열 조립 위험
② 쿼리 빌더    SQL 구조 그대로 + 타입 안전
③ ORM         객체로 다룸, SQL이 감춰짐
```

**학습 단계 권장**: ②를 주력으로 쓰되, **생 SQL을 읽고 쓸 줄 아는 상태**를 유지한다.
③만 쓰면 무슨 쿼리가 나가는지 모르게 되고, 그 순간 성능 문제를 볼 수 없게 된다.

(구체적 도구 선택은 [11-2026 생태계 현황 스냅샷](11-2026-생태계-현황-스냅샷.md) 참조 — 이 영역은 변화가 빠르다.)

### N+1 문제

```ts
// ❌ N+1
const posts = await db.select().from(postsTable);        // 쿼리 1번
for (const post of posts) {
  post.author = await db.select().from(users)            // 쿼리 N번!
                        .where(eq(users.id, post.userId));
}
// 게시글 100개 → 쿼리 101번 → 각 2ms → 202ms
```

```ts
// ✅ 배치
const posts   = await db.select().from(postsTable);
const userIds = [...new Set(posts.map(p => p.userId))];
const authors = await db.select().from(users)
                        .where(inArray(users.id, userIds));   // 쿼리 1번
// 총 2번 → 4ms
```

> **이건 새로운 문제가 아니다.** Stage 2의 "루프 안의 await"와 **정확히 같은 병리**다.
> 원인은 연산량이 아니라 **비용을 항목마다 치른 것**이고, 처방도 같다 — **배치화**.
>
> 이 패턴은 백엔드 전반에 반복해서 나타난다. 이름만 바뀔 뿐이다.
> (N+1 쿼리 / 루프 안의 await / 요청마다 개별 API 호출 …)

### 마이그레이션 — 스키마의 버전 관리

```
문제: 코드는 git으로 버전 관리되는데, DB 스키마는?
      개발 DB엔 컬럼이 있고 운영 DB엔 없으면 배포가 터진다.
```

```
migrations/
  0001_create_users.sql
  0002_add_email_index.sql
  0003_add_posts_table.sql
```

각 파일을 **순서대로, 한 번씩만** 적용한다. DB가 "어디까지 적용했는지"를 기록한다.

**규칙**: 이미 적용된 마이그레이션은 **고치지 않는다.** 새 마이그레이션을 추가한다.
(고치면 이미 적용한 환경과 새로 적용하는 환경의 결과가 달라진다.)

### SQL DB vs NoSQL — 교양 수준 정리

| | 관계형 (PostgreSQL, SQLite) | 문서형 (MongoDB) |
|---|---|---|
| 스키마 | 미리 정함 | 유연 |
| 관계 | JOIN | 앱에서 조합하거나 중첩 |
| 트랜잭션 | 강함 | 제한적/후발 |
| 언제 | **대부분의 경우** | 스키마가 진짜 유동적일 때 |

> **실용적 결론**: 잘 모르겠으면 **PostgreSQL**이다.
> "NoSQL이 더 빠르다"는 대부분 잘못 설계된 관계형 DB와 비교한 것이다.
> 학습·개인 프로젝트에는 **SQLite**가 세팅 비용 0으로 최고다.

## 이 스테이지의 함정

| 함정 | 교정 |
|---|---|
| ORM만 배우고 SQL을 안 배움 | SQL이 본체, ORM은 편의. 순서가 반대면 성능 문제를 못 본다 |
| 트랜잭션을 "그냥 감싸면 되는 것"으로 이해 | 격리 수준과 락을 알아야 실제로 안전해진다 |
| 트랜잭션 안에서 외부 호출 | 락을 네트워크 시간만큼 잡는다. 금지 |
| 인덱스 남발 | 쓰기 성능이 죽는다. 측정 후 추가 |
| NoSQL로 도망 | 관계형을 모르는 상태에서의 NoSQL은 더 어렵다 |
| 마이그레이션 없이 손으로 스키마 변경 | 운영/개발 불일치. 반드시 파일로 |

## 결론

> DB는 "데이터를 저장하는 곳"이 아니라
> **"여러 명이 동시에 손대도 진실이 하나로 유지되게 만드는 장치"**다.
>
> 저장만이 목적이면 파일로 충분하다. 트랜잭션이 없으면 DB를 쓸 이유의 절반이 사라진다.
>
> 그래서 이 스테이지에서 정말 배워야 할 것은 SQL 문법이 아니라 **동시성**이다.
